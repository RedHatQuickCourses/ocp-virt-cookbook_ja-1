= リアルタイム仮想マシン
:navtitle: リアルタイム VM

== 概要

このチュートリアルでは、OpenShift Virtualization でリアルタイム仮想マシンを構成する方法を示します。リアルタイム仮想マシンは、厳格な、制限されたレイテンシー要件を持つワークロードに最適化されています：5G RAN (vDU/vCU)、工業制御システム、金融のトレードから取引までのリアルタイム処理、高速なテレメトリー。

共有 CPU ホスト上の標準仮想マシンは、他のワークロードからの CPU 窃取によるスケジューリング遅延のピークが数十ミリ秒になる可能性があります。適切に構成されたリアルタイム仮想マシンは、複数のレイヤーのチューニングを組み合わせて、最悪のレイテンシーを低微秒レベルに低下させます。

[cols="2,3,2",options="header"]
|===
|構成層 |ジッター源の除去 |構成場所

|専用 CPU 配置
|他のワークロードからの CPU 窃取
|仮想マシン仕様 (`dedicatedCpuPlacement`)

|エミュレータースレッドの分離
|vCPU コア上の QEMU の家庭用作業
|仮想マシン仕様 (`isolateEmulatorThread`)

|NUMA パススルー
|クロス NUMA メモリーアクセス遅延
|仮想マシン仕様 (`guestMappingPassthrough`)

|ハイパーページ
|TLB ミスとページテーブルウォーク
|仮想マシン仕様 (`memory.hugepages`)

|ホスト上の RT カーネル
|カーネル事前予約遅延、タイマージッター
|パフォーマンスプロファイル (`realTimeKernel`)

|ホスト上の CPU 隔離
|IRQ とカーネルスレッドの干渉
|パフォーマンスプロファイル (`cpu.isolated`)

|ゲスト内スケジューリング
|ゲスト vCPU のスケジューリング優先度
|仮想マシン仕様 (`cpu.realtime`)

|バルーン/グラフィックスの無効化
|不要なデバイス中断
|仮想マシン仕様 (`autoattachMemBalloon: false`)

|===

このチュートリアルは、パフォーマンスシリーズの第 3 部です。xref:cpu-pinning.adoc[専用 CPU 配置] と xref:numa-hugepages.adoc[NUMA/ハイパーページ] チュートリアルから直接構築されます。

== 前提条件

* OpenShift 4.18+ に OpenShift Virtualization オペレーターがインストールされていること (テスト済みバージョン: 4.21)
* クラスター管理者権限 (`system:admin` または等価)
* `oc` と `virtctl` CLI がインストールされていること
* 8 つ以上の物理 CPU コアと 16Gi RAM を持つワーカーノード
* CPU マネージャーの静的ポリシーが有効になっていること (xref:cpu-pinning.adoc[専用 CPU 配置] チュートリアルを参照)
* NUMA トポロジーとハイパーページについての理解 (xref:numa-hugepages.adoc[NUMA とハイパーページ] チュートリアルを参照)
* ノードチューニングオペレーターがインストールされていること (OpenShift でデフォルトでインストールされている)

== リアルタイム要件の理解

仮想化のコンテキストでは、リアルタイムは「制限された最悪レイテンシー」を意味します。平均応答時間の速さではありません。リアルタイム仮想マシンは、負荷を受けても、特定の操作が確定時間枠内で完了することを保証します。

これを実現するには、ホスト カーネル、ハイパーバイザー、ゲスト カーネル、およびアプリケーションの間のすべてのジッター源を除去する必要があります。構成表は、各仮想マシン仕様フィールドがどのジッター源を取り扱うかをマッピングしています。

重要: リアルタイム仮想マシンは、その弱い層に依存します。単一の層を省略する (例えば、RT カーネルを使用するが CPU を隔離しない) は、ジッター源をアクティブに残し、目的を果たさないことを意味します。

== ステップ 1: リアルタイムマシン設定プールの準備

リアルタイムノードは、パフォーマンスプロファイルが RT カーネル、CPU 隔離、ハイパーページ構成を適用するのを妨げないように、専用のマシン設定プールが必要です。

まず、ターゲットワーカーノードに `worker-rt` ロールをラベル付けします:

[source,bash,role=execute]
----
oc label node <node-name> node-role.kubernetes.io/worker-rt=
----

`worker` と `worker-rt` の両方のロールを選択するマシン設定プールを作成します:

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfigPool
metadata:
  name: worker-rt
  labels:
    machineconfiguration.openshift.io/role: worker-rt
spec:
  machineConfigSelector:
    matchExpressions:
    - key: machineconfiguration.openshift.io/role
      operator: In
      values:
      - worker
      - worker-rt
  nodeSelector:
    matchLabels:
      node-role.kubernetes.io/worker-rt: ""
EOF
----

プールが作成され、ノードがリストされていることを確認します:

[source,bash,role=execute]
----
oc get mcp worker-rt
----

期待される出力:

----
NAME        CONFIG   UPDATED   UPDATING   DEGRADED   MACHINECOUNT   READYMACHINECOUNT
worker-rt            True      False      False      1              1
----

== ステップ 2: パフォーマンスプロファイルの適用

パフォーマンスプロファイルリソースは、ノードチューニングオペレーターを構成して、次のことを行います:

* リアルタイムカーネル (`kernel-rt`) のインストール
* 2-7 の CPU をワークロード用に隔離 (これらの CPU 上にカーネルスレッドや IRQ がスケジュールされない)
* 0-1 の CPU をシステムサービス用に予約
* 4 個の 1Gi ハイパーページの事前割り当て
* 遅延を最小限に抑えるためのカーネルブートパラメーターの適用

xref:attachment$realtime-vms/performance-profile.yaml[performance-profile.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: performance.openshift.io/v2
kind: PerformanceProfile
metadata:
  name: realtime-workers
spec:
  cpu:
    isolated: "2-7"
    reserved: "0-1"
  hugepages:
    defaultHugepagesSize: 1G
    pages:
    - size: 1G
      count: 4
  realTimeKernel:
    enabled: true
  additionalKernelArgs:
  - nmi_watchdog=0
  - audit=0
  - mce=off
  - processor.max_cstate=1
  - idle=poll
  - intel_idle.max_cstate=0
  nodeSelector:
    node-role.kubernetes.io/worker-rt: ""
  machineConfigPoolSelector:
    machineconfiguration.openshift.io/role: worker-rt
EOF
----

注: 単一ノード OpenShift (SNO) の場合、ステップ 1 で作成するマシン設定プールをスキップします。代わりに、`nodeSelector` を `node-role.kubernetes.io/master: ""` に、`machineConfigPoolSelector` を `pools.operator.machineconfiguration.openshift.io/master: ""` に設定します。これにより、既存のマスタープールを直接ターゲットにします。`machineConfigPoolSelector` は、MCP の自身の `metadata.labels` にあるラベルと一致する必要があります。MCP の内部的に使用する `machineConfigSelector` では一致する必要はありません。まだ、`worker-rt` ロールラベル (ステップ 1 で `oc label node <node-name> node-role.kubernetes.io/worker-rt=` を実行する必要があります)。VM 仕様がこのラベルを使用するためです。

重要: パフォーマンスプロファイル API は、SI 単位 (`1G`, `2M`) を使用してハイパーページサイズを表現します (`1Gi`, `2Mi`)。これは、VM 仕様で使用される Kubernetes リソース構文 (使用される `1Gi` と `2Mi`) とは異なります。

パフォーマンスプロファイルは、いくつかの基礎となるリソースを生成します: `Tuned` プロファイル、カーネル引数用の `MachineConfig`、CPU マネージャー設定用の `KubeletConfig`。その後、マシン設定オペレーターはターゲットノードを再起動します。

警告: パフォーマンスプロファイルを適用すると、RT カーネルをインストールし、カーネルパラメーターを適用するためにターゲットノードを再起動します。影響を受けるノードでのダウンタイムを計画してください。

ロールアウトを監視します:

[source,bash,role=execute]
----
oc get mcp worker-rt -w
----

SNO の場合、マスタープールを監視します:

[source,bash,role=execute]
----
oc get mcp master -w
----

`UPDATED` が `True` で、`READYMACHINECOUNT` が `MACHINECOUNT` と一致するまで待ちます。これは、RT カーネルをダウンロードし、再起動し、クラスターに再参加するのに 10-20 分かかる場合があります。MCP は、再起動が完了した後も `UPDATING: True` 状態を維持し、ノードが構成を再調整し、クラスターに準備状態を報告するのに時間がかかります。ロールアウトが完全に完了する前に進めないでください。

== ステップ 3: リアルタイムノードの準備ができたことの確認

マシン設定プールのロールアウトが完了した後、ノード構成を確認します。

カーネルバージョンを確認します:

[source,bash,role=execute]
----
oc debug node/<node-name>
----

デバッグシェル内で:

[source,bash,role=execute]
----
chroot /host uname -r
----

カーネルバージョンには `rt` が含まれている必要があります。例えば:

----
5.14.0-427.55.1.el9_4.x86_64+rt
----

CPU 隔離が構成されていることを確認するために、カーネルブートパラメーターを確認します:

[source,bash,role=execute]
----
chroot /host cat /proc/cmdline | grep -o 'isolcpus=managed_irq,[^ ]*'
----

期待される出力:

----
isolcpus=managed_irq,2-7
----

隔離された CPU でのノイズレス (nohz) モードがアクティブであることを確認します:

[source,bash,role=execute]
----
chroot /host cat /sys/devices/system/cpu/nohz_full
----

PerformanceProfile で指定されたものと一致する予期される出力:

----
2-7
----

デフォルトの IRQ 割り当てマスクを確認します:

[source,bash,role=execute]
----
chroot /host cat /proc/irq/default_smp_affinity
----

`managed_irq` モードでは、割り当てマスクがすべての CPU を示す可能性があります。IRQ ステアリングは、`irqbalance` ヒントによって動的に処理されるため、静的割り当てマスクではありません。

ハイパーページ割り当てを確認します:

[source,bash,role=execute]
----
chroot /host grep -i hugepages /proc/meminfo
----

要求されたカウント (1Gi ページの場合: 4) と一致する `HugePages_Total` を確認します。

`kubevirt.io/realtime` ラベルが存在することを確認します:

[source,bash,role=execute]
----
exit
----

[source,bash,role=execute]
----
oc get node <node-name> -L kubevirt.io/realtime
----

KubeVirt は、`kernel.sched_rt_runtime_us` が -1 (リアルタイムスケジューリングを制限なしで) に設定されているノードに `kubevirt.io/realtime` ラベルを自動的に付け加えます。ラベルが欠落している場合は、設定を確認します:

[source,bash,role=execute]
----
oc debug node/<node-name> -- chroot /host sysctl kernel.sched_rt_runtime_us
----

期待される出力:

----
kernel.sched_rt_runtime_us = -1
----

== ステップ 5: リアルタイム仮想マシンのデプロイ

以下の仮想マシンマニフェストは、すべてのリアルタイムチューニング層を組み合わせています。標準の Fedora コンテナディスクを使用します。目的構築された RT ゲストイメージ (リアルタイムカーネルを搭載) は、最適な結果を提供しますが、標準のイメージは、ホスト側の構成を示すためおよび検証するために十分です。

重要な構成フィールド:

* `cpu.model: host-passthrough` -- ゲストにすべてのホスト CPU 機能を公開し、マスクしません
* `cpu.dedicatedCpuPlacement: true` -- vCPU を隔離された物理 CPU にピン固定します
* `cpu.isolateEmulatorThread: true` -- QEMU エミュレータースレッドに専用の CPU を予約します
* `cpu.ioThreadsPolicy: auto` -- I/O スレッドをエミュレータースレッドと共に配置します
* `cpu.numa.guestMappingPassthrough: {}` -- ホスト NUMA トポロジーをゲストにパススルーします
* `cpu.realtime.mask: "0"` -- vCPU 0 を SCHED_FIFO スケジューリングに構成し、ハウスキーピングタスクを処理する vCPU 1 をデフォルトスケジューリングに残します
* `memory.hugepages.pageSize: 1Gi` -- 1Gi ハイパーページプールからメモリーを割り当てます
* `autoattachMemBalloon: false` -- バルーンデバイスを無効にして、メモリー管理中の中断を避けます
* `autoattachGraphicsDevice: false` -- グラフィックスデバイスを無効にして、中断源を減らします

注意: この例のクラウドインイメージのパスワードは、デモ目的でのみ使用されます。実稼働環境では、SSH キー認証または強力で一意のパスワードを使用してください。

xref:attachment$realtime-vms/vm-realtime.yaml[vm-realtime.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: vm-realtime
  namespace: realtime-demo
spec:
  runStrategy: Always
  template:
    metadata:
      labels:
        kubevirt.io/domain: vm-realtime
    spec:
      nodeSelector:
        node-role.kubernetes.io/worker-rt: ""
      domain:
        cpu:
          model: host-passthrough
          sockets: 1
          cores: 2
          threads: 1
          dedicatedCpuPlacement: true
          isolateEmulatorThread: true
          ioThreadsPolicy: auto
          numa:
            guestMappingPassthrough: {}
          realtime:
            mask: "0"
        memory:
          hugepages:
            pageSize: 1Gi
        resources:
          requests:
            memory: 2Gi
            cpu: 2
          limits:
            memory: 2Gi
            cpu: 2
        devices:
          autoattachSerialConsole: true
          autoattachMemBalloon: false
          autoattachGraphicsDevice: false
          disks:
          - name: rootdisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            masquerade: {}
      networks:
      - name: default
        pod: {}
      terminationGracePeriodSeconds: 0
      volumes:
      - name: rootdisk
        containerDisk:
          image: quay.io/containerdisks/fedora:latest
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            password: redhat
            chpasswd:
              expire: false
EOF
----

実行中の状態に到達するまで待ちます:

[source,bash,role=execute]
----
oc get vm vm-realtime -n realtime-demo -w
----

期待される出力:

----
NAME          AGE   STATUS    READY
vm-realtime   90s   Running   True
----

注: VM は、`kubevirt.io/realtime` と `cpumanager=true` のラベルを持つノードにのみスケジュールされます。KubeVirt は、VM 仕様で `realtime` を指定すると、これらのセレクターを自動的に追加します。VM が `Scheduling` 状態に残る場合は、ターゲットノードが両方のラベルを持っているかを確認してください。

== ステップ 6: ホスト上のリアルタイムチューニングの確認

== CPU 固定の確認

実行中の VM のノードを見つけます:

[source,bash,role=execute]
----
oc get vmi vm-realtime -n realtime-demo -o wide
----

そのノードでデバッグセッションを開きます:

[source,bash,role=execute]
----
oc debug node/<node-name>
----

CPU マネージャーの状態を確認して、割り当てられた物理 CPU を確認します:

[source,bash,role=execute]
----
chroot /host cat /var/lib/kubelet/cpu_manager_state
----

`entries` マップは、`compute` コンテナーに 2 vCPU ID プラス 1 エミュレータースレッド CPU (3 つの総数) を示す必要があります。すべての割り当てされた CPU は、PerformanceProfile で指定された隔離範囲内 (2-7) にある必要があります。

== ハイパーページの確認

[source,bash,role=execute]
----
chroot /host grep HugePages /proc/meminfo
----

VM が 2 ページ (2Gi VM および 1Gi ハイパーページ) を消費した場合、`HugePages_Free` は `HugePages_Total` より少なくなければなりません。

注: `guestMappingPassthrough` を使用する場合、各 NUMA ノードで vCPU がスケジュールされる場合にハイパーページが利用可能であることを確認してください。PerformanceProfile がハイパーページを NUMA ノード間で均等に配分する場合、要求されたメモリーの量が単一のノードの割り当て量を超えると、割り当てに失敗する可能性があります。VM のメモリー要求量は、各 NUMA ノードのハイパーページ割り当ての割り当て量と同じに保つ必要があります。

デバッグシェルを終了します:

[source,bash,role=execute]
----
exit
----

== ステップ 7: VM 内のリアルタイム構成の確認

コンソールに接続します:

[source,bash,role=execute]
----
virtctl console vm-realtime -n realtime-demo
----

`fedora` ユーザーとして `redhat` パスワードでログインします。

== カーネルタイプの確認

[source,bash,role=execute]
----
uname -r
----

ゲストが標準カーネル ( `rt` シフィックスがない) を実行している場合、ホストレベルで vCPU のリアルタイムスケジューリングはまだアクティブです。目的構築されたゲストイメージに RT カーネルを搭載することで、追加のイングストレス予約が提供され、生産ワークロードのための推奨事項です。

== CPU コアの確認

[source,bash,role=execute]
----
lscpu | grep -E "^CPU\(s\)|^Socket|^Core|^Thread"
----

期待される出力:

----
CPU(s):                  2
Thread(s) per core:      1
Core(s) per socket:      2
Socket(s):               1
----

== スケジューリングポリシーの確認 (ホストの視点)

ゲストコンソールから離脱します (`Ctrl+]`)。

ホストレベルのリアルタイムスケジューリングは、KubeVirt が QEMU vCPU スレッドで構成されます。これをホストデバッグセッションで確認できます。VM のノードでデバッグセッションを開きます:

[source,bash,role=execute]
----
oc debug node/<node-name>
----

QEMU プロセス ID を見つけます:

[source,bash,role=execute]
----
chroot /host pgrep qemu-kvm
----

その PID を使用して、そのプロセスのすべてのスレッドを表示します ( `<qemu-pid>` を前のコマンドからの PID に置き換えます):

[source,bash,role=execute]
----
chroot /host ps -T -o tid,cls,rtprio,comm -p <qemu-pid>
----

期待される出力:

----
    TID CLS RTPRIO COMMAND
  70898  TS      - qemu-kvm
  70900  TS      - call_rcu
  70901  TS      - TC tc-ram-node0
  70941  TS      - IO mon_iothread
  70942  FF      1 CPU 0/KVM
  70943  TS      - CPU 1/KVM
----

`CLS` 列は、スケジューリングクラスを示します: `FF` は `SCHED_FIFO`、`TS` は `SCHED_OTHER` です。`RTPRIO` 列は、リアルタイム優先度を示します。スレッド `CPU 0/KVM` は `FF` で優先度 `1` を示し、`realtime.mask: "0"` で指定されたように `SCHED_FIFO` でスケジュールされていることを確認します。スレッド `CPU 1/KVM` は `TS` を示し、ハウスキーピングタスクのためにデフォルトスケジューリングに残されていることを確認します。

デバッグシェルを終了します:

[source,bash,role=execute]
----
exit
----

== ステップ 8: ラテンシーの測定とベースラインの設定

正しいラテンシーベースラインを得るために、`cyclictest` は標準的なツールです。これは、リアルタイムスレッドの最悪の起動遅延を測定します。

コンソールに接続します:

[source,bash,role=execute]
----
virtctl console vm-realtime -n realtime-demo
----

ログインし、 `realtime-tests` ( `cyclictest` を提供) をインストールします:

[source,bash,role=execute]
----
sudo dnf install -y realtime-tests
----

3 つのスレッドで 60 秒間の短い `cyclictest` を実行して、ラテンシーを測定します:

[source,bash,role=execute]
----
sudo cyclictest --mlockall --smp --priority=80 --interval=200 --distance=0 --duration=60
----

例出力:

----
# /dev/cpu_dma_latency set to 0us
policy: fifo: loadavg: 0.92 0.42 0.18 1/148 1599

T: 0 ( 1597) P:80 I:200 C: 156868 Min:      3 Act:    5 Avg:  191 Max:    9501
