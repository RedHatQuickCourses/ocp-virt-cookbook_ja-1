= 仮想マシンの内部 DNS 解決
:navtitle: VM の内部 DNS

== 概要

このチュートリアルでは、OpenShift Virtualization で仮想マシンを安定した DNS 名でクラスター内で検出可能にする方法を示します。Kubernetes サービスが仮想マシンに対する DNS ベースのサービス検出を可能にし、Pod-to-VM と VM-to-VM の通信に必要な IP アドレスのハードコーディングを避けることができます。

== 学習目標

* Kubernetes サービスを使用して仮想マシンを公開する方法
* 仮想マシンの DNS 解決をテストする方法
* OpenShift Virtualization で仮想マシンの DNS クエリがどのように解決されるか

== 前提条件

* OpenShift 4.18+ で OpenShift Virtualization オペレーターがインストールされている
* CLI ツールがインストールされている: `oc`, `virtctl`
* Kubernetes サービスと DNS についての基本的な理解

== 第1部: デプロイ、公開、テスト

=== ステップ1: ネームスペースの作成

このチュートリアルのために専用のネームスペースを作成します:

[source,bash,role=execute]
----
oc new-project myapp
----

=== ステップ2: 仮想マシンのデプロイ

Fedora 仮想マシンをデプロイします:

警告: この例では、cloud-init パスワードはデモ目的で使用されています。実際の生産環境では、強力で一意のパスワードまたは SSH キー認証を使用してください。

xref:attachment$internal-dns-for-vms/fedora-app-vm.yaml[fedora-app-vm.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-app-vm
  namespace: myapp
  labels:
    app: myapp
spec:
  runStrategy: Always
  dataVolumeTemplates:
    - metadata:
        name: fedora-app-vm-volume
      spec:
        sourceRef:
          kind: DataSource
          name: fedora
          namespace: openshift-virtualization-os-images
        storage:
          resources:
            requests:
              storage: 30Gi
  template:
    metadata:
      labels:
        kubevirt.io/domain: fedora-app-vm
        app: myapp
    spec:
      domain:
        devices:
          disks:
            - disk:
                bus: virtio
              name: rootdisk
            - disk:
                bus: virtio
              name: cloudinitdisk
          interfaces:
            - name: default
              masquerade: {}
        resources:
          requests:
            memory: 2Gi
      networks:
        - name: default
          pod: {}
      volumes:
        - dataVolume:
            name: fedora-app-vm-volume
          name: rootdisk
        - cloudInitNoCloud:
            userData: |-
              #cloud-config
              user: fedora
              password: fedora
              chpasswd: { expire: false }
              ssh_pwauth: true
          name: cloudinitdisk
EOF
----

VM が実行中になるまで待ちます:

[source,bash]
----
watch 'oc get vm,vmi,dv -n myapp'
----

VM が `Running` と `Ready: True` になったら、`Ctrl+C` を押します。

=== ステップ3: VM に対するサービスの作成

VM をターゲットとするヘッドレス サービスを作成します:

xref:attachment$internal-dns-for-vms/headless-service.yaml[headless-service.yaml をダウンロード]

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
  namespace: myapp
spec:
  clusterIP: None
  selector:
    kubevirt.io/domain: fedora-app-vm
  ports:
    - name: ssh
      protocol: TCP
      port: 22
      targetPort: 22
EOF
----

サービスとその EndpointSlice を確認します:

[source,bash,role=execute]
----
oc get svc myapp-service -n myapp
oc get endpointslice -n myapp -l kubernetes.io/service-name=myapp-service
----

期待される出力:

サービスは `TYPE: ClusterIP` で `CLUSTER-IP: None` (ヘッドレスサービス) である必要があります。

EndpointSlice は VM の Pod IP アドレスを ENDPOINTS 列に表示する必要があります。

=== ステップ4: DNS 解決と接続をテスト

DNS 解決と接続をテストするためのデバッグ Podを作成します:

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: debug-dns
  namespace: myapp
spec:
  containers:
  - name: debug
    image: registry.access.redhat.com/ubi9/ubi
    command: ["sleep", "3600"]
    securityContext:
      allowPrivilegeEscalation: false
      runAsNonRoot: true
      runAsUser: 1000
      capabilities:
        drop:
        - ALL
      seccompProfile:
        type: RuntimeDefault
  restartPolicy: Never
EOF

# ポッドが準備ができたことを待ちます
oc wait --for=condition=Ready pod/debug-dns -n myapp --timeout=60s

# DNS 解決をテスト
oc exec -n myapp debug-dns -- getent hosts myapp-service.myapp.svc.cluster.local

# クリーンアップ
oc delete pod debug-dns -n myapp
----

期待される結果:

* DNS 解決が VM の Pod IP アドレス (例: `10.128.0.169 myapp-service.myapp.svc.cluster.local`) を返します

これにより、サービスの DNS 名が VM の IP アドレスに正しく解決されることが確認されます。

== 第2部: これがどのように機能するかの理解

=== DNS 解決フロー

クライアント (別の VM または Pod) がサービスの DNS 名を使用して接続する必要がある場合、以下のフローが発生します:

[source,text]
----
┌───────────────────────────────────────────────────────────────┐
│ OpenShift クラスター                                             │
│                                                               │
│  [クライアント VM/Pod]                                              │
│         │                                                     │
│         │ 1. DNS クエリ: myapp-service.myapp.svc...            │
│         │                                                     │
│         ↓                                                     │
│  [/etc/resolv.conf]                                           │
│   nameserver 172.30.0.10                                      │
│   search myapp.svc.cluster.local svc.cluster.local            │
│         │                                                     │
│         │ 2. クエリを DNS サーバーに送信                       │
│         ↓                                                     │
│  ┌──────────────┐                                             │
│  │   CoreDNS    │                                             │
│  │ 172.30.0.10  │                                             │
│  └──────┬───────┘                                             │
│         │                                                     │
│         │ 3. サービス記録の検索                                 │
│         ↓                                                     │
│  ┌──────────────────────────────────────────────┐             │
│  │ Service: myapp-service                       │             │
│  │ Type: Headless                               │             │
│  │ Selector: kubevirt.io/domain=fedora-app-vm   │             │
│  └──────┬───────────────────────────────────────┘             │
│         │                                                     │
│         │ 4. VM Pod IP を返す                               │
│         ↓                                                     │
│  [クライアントは: 10.128.2.45]                               │
│         │                                                     │
│         │ 5. VM に接続                                     │
│         ↓                                                     │
│  ┌──────────────────────┐                                     │
│  │  fedora-app-vm       │                                     │
│  │  IP: 10.128.2.45     │                                     │
│  └──────────────────────┘                                     │
│                                                               │
└───────────────────────────────────────────────────────────────┘
----

=== ステップバイステップの説明

**ステップ1: DNS クエリの開始**

接続先として `myapp-service.myapp.svc.cluster.local` を必要とするクライアント (別の VM または Pod) は、DNS クエリを開始します。

**ステップ2: クエリを CoreDNS に送信**

クライアントは、`/etc/resolv.conf` ファイルを読み取り、クラスター DNS サービス (通常は `172.30.0.10`) を指すことを確認します。DNS クエリは CoreDNS に送信されます。

**ステップ3: CoreDNS がサービス記録を検索**

CoreDNS は、クラスター内のすべての Kubernetes サービスのデータベースを維持します。CoreDNS は、名前 `myapp-service` を持つ `myapp` ネームスペースにあるサービスを検索します。

**ステップ4: サービスが VM Pod IP を返す**

ヘッドレスサービス (`clusterIP: None`) であるため、CoreDNS は EndpointSlice から VM の Pod IP を返します。通常の ClusterIP サービスの場合、CoreDNS はサービスの仮想 IP を返します。

**ステップ5: 接続が確立**

クライアントは IP アドレスを受け取り、VM に直接接続します。

=== 要約

* サービスが必要である: 仮想マシンは CoreDNS に自動的に登録されません。サービスのみが DNS 記録を作成します。

* VM の名前は DNS 名ではありません: VM の `metadata.name` は Kubernetes リソースの識別子であり、ネットワークアクセスには使用できません。

* サービスの名前が DNS 名になります: サービスの `metadata.name` は `<service-name>.<namespace>.svc.cluster.local` と CoreDNS に登録されます。

* ヘッドレスサービスが VM IP を返します: ヘッドレスサービス (`clusterIP: None`) は、VM の Pod IP を直接返し、VM-to-VM の直接通信を可能にします。

* 通常のサービスはロードバランシングを提供します: 通常の ClusterIP サービスは、安定した仮想 IP を提供し、複数の VM 間でトラフィックを分散できます。

* 検索ドメインが短名を可能にします: `/etc/resolv.conf` の `search` ドメインは、同じネームスペース内のサービスにアクセスする際に短い名前 (例: `myapp-service`) を使用できるようにします。

== 第3部: 参照

=== 確認コマンド

**サービスと EndpointSlice を確認:**

[source,bash,role=execute]
----
# サービスの詳細を表示
oc get svc myapp-service -n myapp

# EndpointSlice (VM Pod IP を表示する必要があります) を確認
oc get endpointslice -n myapp -l kubernetes.io/service-name=myapp-service

# サービスの詳細を表示
oc describe svc myapp-service -n myapp
----

**DNS 解決と接続をテスト:**

[source,bash,role=execute]
----
# デバッグ Pod (既に存在しない場合は作成)
oc apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: debug-dns
  namespace: myapp
spec:
  containers:
  - name: debug
    image: registry.access.redhat.com/ubi9/ubi
    command: ["sleep", "3600"]
    securityContext:
      allowPrivilegeEscalation: false
      runAsNonRoot: true
      runAsUser: 1000
      capabilities:
        drop:
        - ALL
      seccompProfile:
        type: RuntimeDefault
  restartPolicy: Never
EOF

# ポッドが準備ができたことを待ちます
oc wait --for=condition=Ready pod/debug-dns -n myapp --timeout=60s

# DNS 解決をテスト
oc exec -n myapp debug-dns -- getent hosts myapp-service.myapp.svc.cluster.local

# 完了後にクリーンアップ
oc delete pod debug-dns -n myapp
----

**VM のラベルを確認:**

[source,bash,role=execute]
----
# VM のラベルがサービスのセレクターと一致するかを確認
oc get vm fedora-app-vm -n myapp -o jsonpath='{.metadata.labels}'
oc get svc myapp-service -n myapp -o jsonpath='{.spec.selector}'
----

=== 一般的な問題

**問題: DNS 解決が結果を返しません**

**診断:**

[source,bash,role=execute]
----
oc get svc myapp-service -n myapp
oc get endpointslice -n myapp -l kubernetes.io/service-name=myapp-service
----

**一般的な原因:**

* サービスのセレクターが VM の `kubevirt.io/domain` ラベルと一致していません
* VM が実行していない (EndpointSlice エントリーがない)
* 間違ったネームスペース

**解決策:**

1. サービスのセレクターが VM の `kubevirt.io/domain` ラベルと正確に一致することを確認します
2. VM が実行されていることを確認します: `oc get vmi -n myapp`
3. サービスと VM が同じネームスペースにあることを確認します

**問題: クロスネームスペースの DNS が失敗**

**診断:**

[source,bash,role=execute]
----
# 異なるネームスペースでデバッグ Podを作成
oc new-project test-namespace 2>/dev/null || true

oc apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: debug-cross-ns
  namespace: test-namespace
spec:
  containers:
  - name: debug
    image: registry.access.redhat.com/ubi9/ubi
    command: ["sleep", "300"]
    securityContext:
      allowPrivilegeEscalation: false
      runAsNonRoot: true
      runAsUser: 1000
      capabilities:
        drop:
        - ALL
      seccompProfile:
        type: RuntimeDefault
  restartPolicy: Never
EOF

# 待ち、テスト
oc wait --for=condition=Ready pod/debug-cross-ns -n test-namespace --timeout=60s
oc exec -n test-namespace debug-cross-ns -- \
  getent hosts myapp-service.myapp.svc.cluster.local

# クリーンアップ
oc delete pod debug-cross-ns -n test-namespace
----

**一般的な原因:**

* FQDN ではなく短い名前を使用している
* ネットワークポリシーがクロスネームスペースのトラフィックをブロックしている

**解決策:**

1. 常に FQDN を使用 (例: `<service>.<namespace>.svc.cluster.local>`)
2. 厳しい NetworkPolicy を確認します

== 要約

このチュートリアルで学んだこと:

* ヘッドレス Kubernetes サービスを使用して仮想マシンを DNS ベースの検出に公開する方法
* CoreDNS がクラスター内でサービス名を VM Pod IP アドレスにどのように解決するか
* ヘッドレスサービス (直接 VM IP) と ClusterIP サービス (仮想 IP とロードバランシング) の違い
* `getent hosts` コマンドを使用して FQDN 形式を使用してデバッグ Podから DNS 解決をテストする方法
* DNS とサービスに関するキー概念 (サービスが必要である理由や検索ドメインが短名解決を可能にする方法)
