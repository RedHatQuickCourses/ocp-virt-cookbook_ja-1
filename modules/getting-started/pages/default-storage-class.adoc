= デフォルトストレージクラスの構成
:navtitle: ストレージ設定

== 概要

このチュートリアルでは、OpenShift Virtualization にデフォルトのストレージクラスを設定する方法をご案内します。デフォルトのストレージクラスは、ビルトインのDataSourceイメージ（Fedora、RHEL、CentOS Stream、Windows）からブートソースを自動的にプロビジョニングするために必要です。デフォルトのストレージクラスが設定されていない場合、これらのDataSourceは保留中になり、仮想マシンの作成ができません。デフォルトを設定することで、ストレージが明示的に定義されていない場合にも、仮想マシンとデータボリュームが指定されたストレージを使用するように自動的に設定されるため、デプロイと管理が簡素化されます。

テスト済みバージョン：
----
OpenShift 4.19、4.20
----

== 前提条件

始めるには、以下のものが必要です。

* **OpenShift クラスター**：OpenShift Virtualization がインストールされた OpenShift クラスター
* **クラスター管理者アクセス**：クラスター管理者権限を持ったクラスターへのアクセス
* **ストレージクラス**：クラスター内に少なくとも 1 つのストレージクラスが構成されている
* **ストレージプロビジョナー**：動的プロビジョニングをサポートするストレージプロビジョナー

== ステップ 1：利用可能なストレージクラスの表示

まず、クラスター内で既に利用可能なストレージクラスを確認し、既にデフォルトのストレージクラスが設定されているかどうかを確認します。

[source,bash,role=execute]
----
oc get sc
----

このコマンドは、名前とプロビジョナーを表示するすべてのストレージクラスをリスト表示します。名前に `(default)` が付いているストレージクラスは、現在デフォルトとして設定されています。

既にデフォルトのストレージクラスが設定されていない例の出力：
----
NAME                    PROVISIONER                             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
ceph-rbd-sc             rbd.csi.ceph.com                        Delete          Immediate              true                   2d
hostpath-provisioner    kubevirt.io/hostpath-provisioner        Delete          WaitForFirstConsumer   false                  1d
local-storage           kubernetes.io/no-provisioner            Delete          WaitForFirstConsumer   false                  1d
----

既にデフォルトのストレージクラスが設定されている例の出力：
----
NAME                    PROVISIONER                             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
ceph-rbd-sc (default)   rbd.csi.ceph.com                        Delete          Immediate              true                   2d
hostpath-provisioner    kubevirt.io/hostpath-provisioner        Delete          WaitForFirstConsumer   false                  1d
local-storage           kubernetes.io/no-provisioner            Delete          WaitForFirstConsumer   false                  1d
----

以下の点に注意してください。

. デフォルトとして設定するストレージクラスの名前
. 既にデフォルトのストレージクラスが設定されているかどうか

== ステップ 2：既存のデフォルトのストレージクラスの削除（必要に応じて）

既にデフォルトのストレージクラスが設定されている場合は、まずそのアノテーションを削除する必要があります。クラスターは 1 つのデフォルトのストレージクラスのみを持つことができます。

ストレージクラスからデフォルトのアノテーションを削除するには：

[source,bash,role=execute]
----
oc annotate sc <the-old-default-storage-class-name> storageclass.kubernetes.io/is-default-class-
----

注：アノテーションキーの末尾にあるハイフンがアノテーションを削除します。

== ステップ 3：ストレージクラスのアノテーション

選択したストレージクラスにアノテーションを追加します。このアノテーションは、この特定のストレージクラスがデフォルトであることを OpenShift に通知します。

アノテーションは、`storageclass.kubernetes.io/is-default-class: "true"` です。

[source,bash,role=execute]
----
oc annotate sc <your-storage-class-name> storageclass.kubernetes.io/is-default-class="true"
----

例えば、ストレージクラスが `ceph-rbd-sc` である場合：

[source,bash,role=execute]
----
oc annotate sc ceph-rbd-sc storageclass.kubernetes.io/is-default-class="true"
----

== ステップ 4：デフォルトのストレージクラスの確認

アノテーションを適用した後、変更が有効になったことを確認します：

[source,bash,role=execute]
----
oc get sc
----

デフォルトを設定した後の例の出力：
----
NAME                    PROVISIONER                             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
ceph-rbd-sc (default)   rbd.csi.ceph.com                        Delete          Immediate              true                   2d
hostpath-provisioner    kubevirt.io/hostpath-provisioner        Delete          WaitForFirstConsumer   false                  1d
local-storage           kubernetes.io/no-provisioner            Delete          WaitForFirstConsumer   false                  1d
----

選択したストレージクラスの名前の隣に `(default)` が表示されていることを確認してください。

== ステップ 5：デフォルトのストレージクラスのテスト

デフォルトのストレージクラスが正しく機能していることを確認するための簡単なテストを作成します。ストレージクラスを指定せずに PVCを作成します：

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: test-default-storage
  namespace: default
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
EOF
----

次に、PVC が正しいストレージクラスに作成されたかどうかを確認します：

[source,bash,role=execute]
----
oc get pvc test-default-storage -n default
----

例の出力：

----
NAME                   STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   AGE
test-default-storage   Pending                                      lvms-vg1       0s
----

PVC は `Pending` 状態にあるが、デフォルトのストレージクラス (`lvms-vg1` など) に自動的に割り当てられています。

=== ボリュームバインディングモードの理解

PVC が `Pending` 状態にあるのは、大多数のストレージクラスが `WaitForFirstConsumer` ボリュームバインディングモードを使用するためです。これは予期される動作です。

[cols="1,2,2"]
|===
|ボリュームバインディングモード |動作 |使用ケース

|`WaitForFirstConsumer`
|Pod が PVC をスケジュールされたときに PV がプロビジョニングされる
|Pod と同じノードにストレージが作成されることを保証し、トポロジー認識ストレージ（LVM、ローカルストレージ）に適しています

|`Immediate`
|PVC が作成されたときに PV がプロビジョニングされる
|ネットワークアタッチストレージ（NFS、Ceph、クラウドストレージ）から任意のノードにアクセスできるため、適しています
|===

ストレージクラスのバインディングモードを確認するには：

[source,bash,role=execute]
----
oc get sc <storage-class-name> -o jsonpath='{.volumeBindingMode}'
----

== ステップ 6：デフォルトのストレージクラスを使用した VM のデプロイ

今度は、データボリューム用にデフォルトのストレージクラスを自動的に使用する仮想マシンをデプロイできます：

[source,bash,role=execute]
----
oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: test-vm-default-storage
  namespace: default
spec:
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: test-vm-disk
    spec:
      pvc:
        accessModes:
        - ReadWriteOnce
        resources:
          requests:
            storage: 10Gi
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
  template:
    metadata:
      labels:
        kubevirt.io/vm: test-vm-default-storage
    spec:
      domain:
        devices:
          disks:
          - name: datavolumedisk
            disk:
              bus: virtio
        resources:
          requests:
            memory: 1Gi
            cpu: 1
      networks:
      - name: default
        pod: {}
      volumes:
      - name: datavolumedisk
        dataVolume:
          name: test-vm-disk
EOF
----

== トラブルシューティング

=== 一般的な問題

==== 複数のデフォルトのストレージクラス

複数のストレージクラスがデフォルトとしてマークされている場合は、すべてのストレージクラスからアノテーションを削除してください：

[source,bash,role=execute]
----
oc get sc -o custom-columns=NAME:.metadata.name,DEFAULT:.metadata.annotations.storageclass\.kubernetes\.io/is-default-class
----

==== ストレージクラスが見つからない

ストレージクラスをアノテーションする前に、その存在を確認してください：

[source,bash,role=execute]
----
oc get sc <storage-class-name>
----

==== パーミッションが拒否された

クラスター管理者権限を持っていることを確認してください：

[source,bash,role=execute]
----
oc auth can-i create storageclass --all-namespaces
----

=== 確認コマンド

すべてのストレージクラスとそのデフォルトステータスを確認します：

[source,bash,role=execute]
----
oc get sc -o wide
----

特定のストレージクラスの詳細情報を表示します：

[source,bash,role=execute]
----
oc describe sc <storage-class-name>
----

すべての名前空間の PVC とそのストレージクラスをリスト表示します：

[source,bash,role=execute]
----
oc get pvc --all-namespaces -o custom-columns=NAMESPACE:.metadata.namespace,NAME:.metadata.name,STORAGECLASS:.spec.storageClassName
----

== ベストプラクティス

. **適切なストレージの選択**：VM に必要なパフォーマンスと機能（例：高速 I/O、スナップショット、暗号化）を提供するストレージクラスを選択します。

. **ワークロードの要件を考慮する**：異なるワークロードには異なるストレージ特性が必要になる場合があります。重要なワークロードには特定のストレージクラスを使用することを検討してください。

. **ストレージ使用量の監視**：ストレージ使用量とパフォーマンスを定期的に監視して、デフォルトのストレージクラスが必要に満たしていることを確認します。

. **選択を文書化する**：将来の参照やトラブルシューティングのために、特定のストレージクラスをデフォルトとして選択した理由を文書化します。

== 要約

このチュートリアルで学んだこと：

* 利用可能なストレージクラスを表示し、既存のデフォルトを特定する方法
* `storageclass.kubernetes.io/is-default-class` アノテーションを使用してストレージクラスをデフォルトとしてアノテーションする方法
* デフォルトのストレージクラスが正しく機能していることを確認するためのテスト PVC と VM のデプロイ方法
* OpenShift Virtualization でデフォルトのストレージクラスを選択および管理するためのベストプラクティス
