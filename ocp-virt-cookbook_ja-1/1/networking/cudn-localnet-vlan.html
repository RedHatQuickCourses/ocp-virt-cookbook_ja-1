  • CUDN を使用したローカルネットワーク VLAN

クラスター用定義されたネットワーク (CUDN) を使用した OpenShift ビジュアライゼーションのローカルネットワーク VLAN の設定

このチュートリアルでは、OpenShift ビジュアライゼーションのクラスター用定義されたネットワーク (CUDN) を使用してローカルネットワーク VLAN を設定する方法を示します。このアプローチにより、VLAN セグメンテーションを複数の名前空間で実行し、仮想マシン (VM) が特定の VLAN タグ付きネットワークに直接アクセスできるようになり、ポッドネットワーク接続を維持します。CUDN は、中央集権的な管理と名前空間間のネットワーク可用性を提供し、VLAN ベースのネットワークインフラストラクチャーを必要とするエンタープライズ環境に適しています。

前提条件

  • NMState Operator: クラスター内でインストールおよび実行されている必要があります

  • NMState CR: オペレーターが実行されている後に nmstate インスタンスを作成する必要があります

  • ワーカーノード: ブリッジマッピングがワーカーノードに適用されます

  • VLAN インフラストラクチャー: 物理ネットワークインフラストラクチャーが VLAN タグ付けをサポートする必要があります

  • スイッチ構成: OpenShift ワーカーノードに接続されている物理スイッチポートが トランク モード で構成されている必要があります。これにより、複数の VLAN タグ付きトラフィックストリームを処理できます

テスト済みバージョン:

OpenShift 4.19

ネットワーク構成要件

物理スイッチ構成: トランク モード

OpenShift ワーカーノードに接続されている物理スイッチポートは、ローカルネットワーク VLAN の設定が正しく動作するために トランク モード で構成する必要があります:

  • トランク モード: スイッチポートが複数の VLAN のトラフィックを運ぶことを許可します

  • VLAN タグ付け: 各 VLAN は、一意の VLAN ID (例: 100、200、300) で識別されます

  • 複数のネットワーク: 複数の VLAN ベースのセカンダリー ネットワークをサポートするために不可欠です

ステップ 1: VM 名前空間の作成

ローカルネットワーク VLAN 設定を使用する VM に対して名前空間を作成します。CUDN は、同じネットワークを複数の名前空間で使用できるようにします。

# 最初の名前空間の作成
oc apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: vm-guests-vlan-prod
EOF

# 2 番目の名前空間の作成
oc apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: vm-guests-vlan-dev
EOF

ステップ 2: 追加の NIC 上のカスタム OVS ブリッジの作成

まず、追加の物理ネットワークインターフェース (NIC) 上にカスタム OVS ブリッジを作成して、VLAN トラフィックを主クラスター ネットワークから分離します。

oc apply -f - <<EOF
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: br-vlan-creation
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ''
  desiredState:
    interfaces:
    - name: enp11s0f0np0  # 物理インターフェース名を置き換えてください
      type: ethernet
      state: up
      ipv4:
        enabled: false
      ipv6:
        enabled: false
      description: "VLAN ローカルネットワーク用の物理インターフェース"
    - name: br-vlan
      type: ovs-bridge
      state: up
      bridge:
        port:
        - name: enp11s0f0np0
      description: "VLAN トラフィック用の OVS ブリッジ"
EOF

注: enp11s0f0np0 を、実際の追加物理インターフェース名に置き換えてください。利用可能なインターフェースを見つけるには、以下のコマンドを実行できます:

oc debug node/<node-name> -- ip link show

ブリッジの作成を確認します:

oc get nncp br-vlan-creation
# 期待される出力: STATUS: Available

ステップ 3: ブリッジマッピングの構成

ローカルネットワーク VLAN 設定で、logical network interface name localnet-vlan を br-vlan OVS ブリッジにマッピングします。このマッピングは、OVN-Kubernetes が VLAN サポートを持つローカルネットワーク トラフィックを処理する物理ブリッジ インターフェースを示します。

oc apply -f - <<EOF
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: cudn-localnet-vlan-bridge-mapping
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ''
  desiredState:
    ovn:
      bridge-mappings:
      - localnet: localnet-vlan
        bridge: br-vlan
        state: present
EOF

ステップ 4: VLAN を持つ CUDN の作成

複数の名前空間でローカルネットワーク トポロジーを設定し、VLAN タグ付けを持つセカンダリー ネットワーク接続を可能にする ClusterUserDefinedNetwork を作成します。

oc apply -f - <<EOF
apiVersion: k8s.ovn.org/v1
kind: ClusterUserDefinedNetwork
metadata:
  name: cudn-localnet-vlan-100
spec:
  namespaceSelector:
    matchExpressions:
    - key: kubernetes.io/metadata.name
      operator: In
      values: ["vm-guests-vlan-prod", "vm-guests-vlan-dev"]
  network:
    topology: Localnet
    localnet:
      role: Secondary
      physicalNetworkName: localnet-vlan
      vlanID: 100
      ipam:
        mode: Disabled
EOF

構成詳細:

  • vlanID: 100: ネットワーク セグメンテーションに VLAN タグ 100 を指定します

  • physicalNetworkName: ブリッジ マッピングの logical network interface name (localnet-vlan) を参照します

  • role: Secondary: このネットワークをセカンダリー ネットワークとして定義します (VM はポッド ネットワークを主ネットワークとして維持します)

  • namespaceSelector: このネットワークを使用できる名前空間を指定します

  • IPAM 無効: 静的 IP アドレスは、cloud-init (静的) または外部 DHCP サーバーによって構成する必要があります

ステップ 5: VLAN ネットワーク接続と静的 IP を持つ VM のデプロイ

1 つの名前空間で、pod ネットワークと VLAN タグ付けされたセカンダリー ネットワークに接続できる VM を作成します。

警告: この例の cloud-init パスワードは、デモのためのものです。実際の環境では、強力で一意のパスワードまたは SSH キー認証を使用してください。

oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-cudn-vlan-vm
  namespace: vm-guests-vlan-prod
  labels:
    app: fedora-cudn-vlan-vm
spec:
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: fedora-cudn-vlan-volume
    spec:
      pvc:
        accessModes:
        - ReadWriteOnce
        resources:
          requests:
            storage: 33Gi
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
  template:
    metadata:
      labels:
        app: fedora-cudn-vlan-vm
    spec:
      domain:
        devices:
          disks:
          - name: datavolumedisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            bridge: {}
          - name: cudn_vlan_network
            bridge: {}
        resources:
          requests:
            memory: 2Gi
            cpu: 1
      networks:
      - name: default
        pod: {}
      - name: cudn_vlan_network
        multus:
          networkName: cudn-localnet-vlan-100
      volumes:
      - name: datavolumedisk
        dataVolume:
          name: fedora-cudn-vlan-volume
      - name: cloudinitdisk
        cloudInitNoCloud:
          networkData: |
            version: 2
            ethernets:
              enp2s0:
                addresses: ["192.168.100.10/24"]
          userData: |
            #cloud-config
            user: fedora
            password: fedora
            chpasswd: { expire: False }
            packages:
            - python3
            - tcpdump
            - net-tools
            write_files:
            - path: /root/index.html
              content: |
                <h1>OpenShift ビジュアライゼーションの CUDN VLAN ネットワークへようこそ!</h1>
                <p>VLAN ID: 100</p>
                <p>静的 IP: 192.168.100.10/24</p>
                <p>ネットワーク: CUDN ローカルネットワーク</p>
                <p>名前空間: vm-guests-vlan-prod</p>
              permissions: '0644'
            runcmd:
            - nohup python3 -m http.server 8080 --directory /root > /dev/null 2>&1 &
EOF

ステップ 7: VLAN ネットワーク接続の検証

VM コンソールにアクセスして、VLAN 設定を持った双方向ネットワーク接続を検証します。

# 生産名前空間の VM の確認
oc get vm -n vm-guests-vlan-prod
oc get vmi -n vm-guests-vlan-prod

# 開発名前空間の VM の確認
oc get vm -n vm-guests-vlan-dev
oc get vmi -n vm-guests-vlan-dev

# 生産名前空間の VM コンソール (cloud-init 認証情報: fedora/fedora) にアクセス
virtctl console -n vm-guests-vlan-prod fedora-cudn-vlan-vm

# 開発名前空間の VM コンソール (cloud-init 認証情報: fedora/fedora) にアクセス
virtctl console -n vm-guests-vlan-dev fedora-cudn-vlan-vm-dev

各 VM コンソール内で、VLAN ネットワーク構成を検証します:

# 主ネットワーク (DHCP を使用した) とセカンダリー ネットワーク (VLAN ネットワーク) のネットワーク インターフェースを確認
ip addr show

# ルートを確認
ip route show

# 期待される出力例 (DHCP を使用した主ネットワーク):
default via 10.128.0.1 dev enp1s0 proto dhcp src 10.128.0.174 metric 100

# 外部接続をテスト
ping 8.8.8.8

# Web サービスが実行中であることを確認
curl localhost:8080

# VLAN インターフェースでの静的 IP 割り当ての確認
ip addr show enp2s0
# 期待される出力: inet 192.168.100.10/24 (生産) または 192.168.100.20/24 (開発)

# VM 間の VLAN インターフェース接続をテスト
# 生産 VM (192.168.100.10) から開発 VM に ping:
ping 192.168.100.20

# tcpdump が利用可能な場合は、VLAN-tagged トラフィックを確認 (セカンダリー インターフェース)
sudo tcpdump -i enp2s0 -nn vlan 100

名前空間間の接続をテスト

異なる名前空間間で VM が共有された CUDN VLAN ネットワークを介して通信できるかどうかを検証します:

# 生産 VM (192.168.100.10) から開発 VM に接続をテスト
ping 192.168.100.20

# 名前空間間で Web サービス接続をテスト
curl 192.168.100.20:8080

# 開発 VM (192.168.100.20) から生産 VM に接続をテスト
ping 192.168.100.10
curl 192.168.100.10:8080

トラブルシューティング

OVS ブリッジおよびブリッジマッピングのトラブルシューティング、および CUDN 固有の検証手順については、OVS ブリッジ検証トラブルシューティング ガイド を参照してください。

一般的な問題

VLAN トラフィックが VM に到達しない

  • 物理スイッチ構成が VLAN タグ付けをサポートしていることを確認してください

  • ワーカーノード上のブリッジマッピング構成を確認してください

  • VLAN ID がネットワークインフラストラクチャーと一致していることを確認してください

  • カスタム OVS ブリッジ (br-vlan) が作成されていることを確認してください: oc get nncp br-vlan-creation

  • 物理インターフェースがブリッジに接続されていることを確認してください

IP アドレス割り当ての問題

  • cloud-init networkData で静的 IP が正しく構成されていることを確認してください

  • インターフェース名が一致していることを確認してください (2 番目のインターフェースの場合: enp2s0)

  • VM 内の cloud-init ログを確認してください: cat /var/log/cloud-init.log

ネットワーク接続の問題

  • 物理ネットワークが構成された VLAN をサポートしていることを確認してください

  • 物理ネットワークから VLAN サブネットへの接続をテストしてください

  • 物理スイッチポートがトランク モードであることを確認してください

名前空間間の問題

  • CUDN 設定の名前空間セレクターを確認してください

  • 両名前空間が値の配列にリストされていることを確認してください

  • NAD がターゲット名前空間で自動作成されていることを確認してください

CUDN 固有の問題

  • CUDN の physicalNetworkName がブリッジマッピングの logical name (localnet-vlan) と一致していることを確認してください

  • 選択された名前空間で CUDN が作成された NAD を確認してください: oc get network-attachment-definitions -A | grep cudn-localnet-vlan-100

  • IPAM モードが正しく (Disabled で静的 IP を cloud-init によって構成する) 設定されていることを確認してください

検証コマンド

# ClusterUserDefinedNetwork の状態を確認
oc get clusteruserdefinednetwork

# ワーカーノード上のブリッジマッピングを確認
oc get nncp cudn-localnet-vlan-bridge-mapping -o yaml

# 両名前空間で VM のネットワーク状態を確認
oc get vmi -n vm-guests-vlan-prod -o yaml | grep -A 10 networks
oc get vmi -n vm-guests-vlan-dev -o yaml | grep -A 10 networks

# OVN-Kubernetes のログを確認してトラブルシューティングを行う
oc logs -n openshift-ovn-kubernetes -l app=ovnkube-node

# 名前空間間で CUDN ネットワーク 接続を確認
oc get network-attachment-definitions -A | grep cudn-localnet-vlan-100
