LVM Storage Operator のインストール

概要

このチュートリアルでは、OpenShift ノードのスパアローカルディスクを使用して LVM Storage operator をインストールおよび構成する方法を示します。LVM operator は、Logical Volume Manager を使用した動的ストレージプロビジョニングを提供し、Single Node OpenShift (SNO) またはバレーメットライブクラスターに適したものです。

前提条件

  • OpenShift 4.18+

  • スパアディスクが利用可能: クラスターノードに少なくとも 1 つのスパアディスク

  • クラスター管理者アクセス: オペレーターのインストールとストレージクラスの作成が必要

  • ディスクに既存のデータがある: forceWipeDevicesAndDestroyAllData パラメーターを使用してディスクを強制的に消去できます

テスト済みバージョン:

OpenShift 4.19, 4.20

重要な警告

注: forceWipeDevicesAndDestroyAllData: true パラメーターを使用すると、指定されたディスク上のすべてのデータが永久的に破棄されます。適切なディスクデバイスパスを使用し、バックアップがあることを確認してから、続行してください。

ステップ 1: 利用可能なディスクの特定

LVM operator を構成する前に、ストレージ用に使用するディスクを特定します。

# ノード名を取得
oc get nodes

# ノード上のすべてのブロックデバイスをリスト表示
oc debug node/<node-name> -- chroot /host lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT,MODEL

例出力:

NAME SIZE TYPE FSTYPE MOUNTPOINT MODEL sda 558.4G disk LVM2_member PERC H730 Adp sdc 558.4G disk PERC H730 Adp |-sdc1 1M part |-sdc2 127M part vfat |-sdc3 384M part ext4 /boot `-sdc4 557.9G part xfs /sysroot

この例では:

* `sda` - 既存の LVM データがあるスパアディスク (このディスクを使用します)
* `sdc` - オペレーティングシステムパーティションを含むシステムディスク (使用しないでください)

既存の LVM 使用状況を確認:

[source,bash,role=execute]

oc debug node/<node-name> — chroot /host pvs

例出力:

PV VG Fmt Attr PSize PFree /dev/sda vg1 lvm2 a-- 558.37g <55.84g

この出力は、`/dev/sda` が既存の LVM 構成を持っており、`forceWipeDevicesAndDestroyAllData: true` を使用すると消去されることを示しています。

永続的な識別子を得るためのディスクデバイスパスを取得:

[source,bash,role=execute]

oc debug node/<node-name> — chroot /host ls -la /dev/disk/by-path/ | grep -E "pci.*scsi"

例出力:

lrwxrwxrwx. 1 root root 9 Oct 6 13:30 pci-0000:03:00.0-scsi-0:2:1:0 → ../../sda

永続的なパス `/dev/disk/by-path/pci-0000:03:00.0-scsi-0:2:1:0` は `/dev/sda` を指します。

== ステップ 2: LVM Storage Operator のインストール

LVM operator の名前空間を作成し、インストールします。

LVM operator の名前空間を作成:

xref:attachment$lvm-operator/lvm-namespace.yaml[lvm-namespace.yaml をダウンロード]

[source,bash,role=execute]

oc apply -f - <<EOF apiVersion: v1 kind: Namespace metadata: name: openshift-lvm-storage EOF

OperatorGroup を作成:

xref:attachment$lvm-operator/lvm-operatorgroup.yaml[lvm-operatorgroup.yaml をダウンロード]

[source,bash,role=execute]

oc apply -f - <<EOF apiVersion: operators.coreos.com/v1 kind: OperatorGroup metadata: name: openshift-lvm-storage-operatorgroup namespace: openshift-lvm-storage spec: targetNamespaces: - openshift-lvm-storage EOF

サブスクリプションを使用してオペレーターをインストール:

xref:attachment$lvm-operator/lvm-subscription.yaml[lvm-subscription.yaml をダウンロード]

[source,bash,role=execute]

oc apply -f - <<EOF apiVersion: operators.coreos.com/v1alpha1 kind: Subscription metadata: name: lvms-operator namespace: openshift-lvm-storage spec: channel: stable-4.20 name: lvms-operator source: redhat-operators sourceNamespace: openshift-marketplace installPlanApproval: Automatic EOF

オペレーターのインストールを確認:

[source,bash,role=execute]

オペレーターのインストールを待機します (2-3 分かかる場合があります)

oc get csv -n openshift-lvm-storage

オペレーターポッドが実行中であることを確認

oc get pods -n openshift-lvm-storage

インストール後の期待出力 (LVMCluster まだ作成していない場合):

NAME READY STATUS RESTARTS AGE lvms-operator-864b6975bb-kxt4t 1/1 Running 0 2m

注: この時点で、`lvms-operator` のみが表示されるはずです。`vg-manager` のポッドは、ステップ 3 で LVMCluster リソースを作成すると自動的に作成されます。

インストール後の期待出力 (ステップ 3 で LVMCluster を作成した場合):

NAME READY STATUS RESTARTS AGE lvms-operator-864b6975bb-kxt4t 1/1 Running 0 5m vg-manager-xxxxx 1/1 Running 0 1m

== ステップ 3: LVMCluster 構成の作成

LVM ストレージバックエンドを構成するための LVMCluster リソースを作成します。ここで、使用するディスクを指定し、`forceWipeDevicesAndDestroyAllData` パラメーターを使用して既存のデータを強制的に消去するかどうかを指定します。

=== 既存データのあるディスクの構成

ディスクに既存のデータまたはパーティション (例: `sda` の例) がある場合は、`forceWipeDevicesAndDestroyAllData: true` を使用します:

注: `forceWipeDevicesAndDestroyAllData: true` を使用すると、指定されたディスク上のすべてのデータが永久的に破棄されます。適切なディスクデバイスパスを使用し、バックアップがあることを確認してから、続行してください。

xref:attachment$lvm-operator/lvmcluster-with-wipe.yaml[lvmcluster-with-wipe.yaml をダウンロード]

[source,bash,role=execute]

oc apply -f - <<EOF apiVersion: lvm.topolvm.io/v1alpha1 kind: LVMCluster metadata: name: lvmcluster namespace: openshift-lvm-storage spec: storage: deviceClasses: - name: vg1 default: true thinPoolConfig: name: thin-pool-1 sizePercent: 90 overprovisionRatio: 10 nodeSelector: nodeSelectorTerms: - matchExpressions: - key: node-role.kubernetes.io/worker operator: In values: - "" deviceSelector: paths: - /dev/sda forceWipeDevicesAndDestroyAllData: true EOF

=== クリーンなディスクの構成

ディスクがクリーンで既存のデータまたはパーティションがない場合は:

xref:attachment$lvm-operator/lvmcluster-clean-disk.yaml[lvmcluster-clean-disk.yaml をダウンロード]

[source,bash,role=execute]

oc apply -f - <<EOF apiVersion: lvm.topolvm.io/v1alpha1 kind: LVMCluster metadata: name: lvmcluster namespace: openshift-lvm-storage spec: storage: deviceClasses: - name: vg1 default: true thinPoolConfig: name: thin-pool-1 sizePercent: 90 overprovisionRatio: 10 nodeSelector: nodeSelectorTerms: - matchExpressions: - key: node-role.kubernetes.io/worker operator: In values: - "" deviceSelector: paths: - /dev/sda EOF

=== 構成パラメーターの説明

* **`name: vg1`**: 作成されるボリュームグループの名前 (ディスクに作成されます)
* **`default: true`**: クラスターのデフォルトのデバイスクラスにします
  ** 目的: デフォルトのデバイスクラスがないときの警告を防ぐ
  ** 効果: 生成される StorageClass を PVC で明示的に指定する必要がなくなります
  ** 注: 複数のデバイスクラスがある場合は、1つだけをデフォルトとしてマークする必要があります
* **`thinPoolConfig`**: 効率的なストレージ使用を可能にするためのスリンクプロビジョニングの構成
  ** `sizePercent: 90`: スリンクプールにディスクの 90% を使用
  ** `overprovisionRatio: 10`: 物理的に利用可能なより多くのストレージを提供できるように 10 倍のオーバープロビジョニングを許可
* **`nodeSelector`**: 構成するノード (この例ではワーカーノード)
* **`deviceSelector.paths`**: 使用する特定のディスクデバイスパス
  ** `/dev/sda`、`/dev/disk/by-path/...`、または `/dev/disk/by-id/...` を使用できます
* **`forceWipeDevicesAndDestroyAllData: true`**: 重要なパラメーター
  ** 目的: ディスク上のすべての既存のパーティションとファイルシステムを削除します
  ** 使用する場合: ディスクに既存のパーティション、ファイルシステム、または LVM 構成がある
  ** 効果: 完全なデータ破棄 - 恢复は不可能
  ** 省略する場合: ディスクが既にクリーンまたは空である

== ステップ 4: LVMCluster の状態を確認

LVMCluster が準備ができているかを確認するためのクイックヘルスチェックを実行します:

[source,bash,role=execute]

LVMCluster の状態を確認

oc get lvmcluster -n openshift-lvm-storage

ボリュームグループが作成されたことを確認

oc debug node/<node-name> — chroot /host vgs

スリンクプールが作成されたことを確認

oc debug node/<node-name> — chroot /host lvs

正常な健全な出力:

* LVMCluster が `STATUS: Ready` で表示される
* VG `vg1` が適切なサイズで表示される
* アクティブな状態のアクティブなスリンクプール `thin-pool-1` が低いデータ/メタ使用率で表示される

詳細な検証手順、期待出力、トラブルシューティングについては、xref:lvm-troubleshooting.adoc[LVM Storage Troubleshooting Guide] を参照してください。

== ステップ 5: デフォルトの StorageClass の設定

LVM operator は自動的に `lvms-vg1` という名前の StorageClass を作成しますが、デフォルトでは設定されていません。OpenShift Virtualization で仮想マシンのディスクや DataVolume を自動的にプロビジョニングするために、デフォルトの StorageClass に設定する必要があります。

=== デフォルトの StorageClass が存在するかどうかを確認

[source,bash,role=execute]

oc get storageclass

期待出力:

NAME PROVISIONER RECLAIMPOLICY VOLUMEBINDINGMODE ALLOWVOLUMEEXPANSION AGE lvms-vg1 topolvm.io Delete WaitForFirstConsumer true 5m

注: `(default)` アノテーションが表示されていないことを確認してください - これはまだデフォルトではありません。

=== デフォルトの StorageClass に設定

クラスターのデフォルトの StorageClass に LVM StorageClass を設定:

[source,bash,role=execute]

oc patch storageclass lvms-vg1 -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'

=== デフォルトのステータスを確認

[source,bash,role=execute]

oc describe sc | grep IsDefaultClass

デフォルト設定後の期待出力:

NAME PROVISIONER RECLAIMPOLICY VOLUMEBINDINGMODE ALLOWVOLUMEEXPANSION AGE lvms-vg1 (default) topolvm.io Delete WaitForFirstConsumer true 6m

`(default)` アノテーションが表示されていることで、これが現在のデフォルト StorageClass であることが確認できます。

=== OpenShift Virtualization での重要性

デフォルトの StorageClass を設定することは、OpenShift Virtualization で重要です:

==== データソース OS イメージがデフォルト StorageClass を必要とする

OpenShift Virtualization のデータソース OS イメージ (Fedora、RHEL、CentOS Stream、Windows) は、デフォルトの StorageClass がないと、ブートソースボリュームを作成できません:

[source,bash,role=execute]

利用可能な OS イメージを確認

oc get datasources -n openshift-virtualization-os-images

デフォルトの StorageClass がないと、これらのデータソースは待機状態に残り、VM の作成には使用できません。

==== VM の作成時の自動 PVC の作成

`dataVolumeTemplates` を使用して VM を作成する場合、`storageClassName` を指定しない場合、デフォルトの StorageClass が使用されます:

[source,yaml]

dataVolumeTemplates: - metadata: name: fedora-volume spec: pvc: accessModes: - ReadWriteOnce resources: requests: storage: 30Gi sourceRef: kind: DataSource name: fedora namespace: openshift-virtualization-os-images

これにより、デフォルト StorageClass (`lvms-vg1`) が自動的に使用されます。明示的に指定する必要はありません。

==== 簡素な VM の作成

デフォルトの StorageClass を設定すると:

* テンプレートはボックスで動作します
* DataSource のクローンが自動的に行われます
* PVC で `storageClassName` を指定する必要はありません
* クラスター全体で一貫したストレージプロビジョニング

==== ブートソースの準備

デフォルトの StorageClass が利用可能な場合、OpenShift Virtualization オペレーターは、一般的な OS イメージのブートソースを自動的に準備できます:

[source,bash,role=execute]

ブートソースの準備状況を確認

oc get DataImportCron -n openshift-virtualization-os-images

=== 代替: VM 内の明示的な StorageClass

デフォルトの StorageClass を設定しない場合は、すべての VM 定義で明示的に指定する必要があります:

[source,yaml]

dataVolumeTemplates: - metadata: name: fedora-volume spec: pvc: accessModes: - ReadWriteOnce resources: requests: storage: 30Gi storageClassName: lvms-vg1 # 明示的に必要 sourceRef: kind: DataSource name: fedora namespace: openshift-virtualization-os-images

注意: OpenShift Virtualization クラスターでは、常にデフォルト StorageClass を設定して、操作を簡素化し、自動ブートソースプロビジョニングを有効にしてください。

== ステップ 6: ストレージプロビジョニングのテスト

PVC を作成して、ストレージプロビジョニングが正常に機能するかを確認します:

xref:attachment$lvm-operator/test-pvc.yaml[test-pvc.yaml をダウンロード]

[source,bash,role=execute]

oc apply -f - <<EOF apiVersion: v1 kind: PersistentVolumeClaim metadata: name: lvms-test-pvc namespace: default spec: accessModes: - ReadWriteOnce resources: requests: storage: 10Gi storageClassName: lvms-vg1 EOF

テスト用のポッドを作成:

xref:attachment$lvm-operator/test-pod.yaml[test-pod.yaml をダウンロード]

[source,bash,role=execute]

oc apply -f - <<EOF apiVersion: v1 kind: Pod metadata: name: lvms-test-pod namespace: default spec: containers: - name: test image: registry.access.redhat.com/ubi9/ubi:latest command: ["sleep", "3600"] volumeMounts: - name: data mountPath: /data volumes: - name: data persistentVolumeClaim: claimName: lvms-test-pvc EOF

PVC がバインドされているかを確認:

[source,bash,role=execute]

oc get pvc lvms-test-pvc

"Bound" と表示される STATUS が期待される

ポッドが実行中でストレージに書き込みできるかを確認:

[source,bash,role=execute]

ポッドが開始されるまで待機

oc wait --for=condition=Ready pod/lvms-test-pod --timeout=120s

ボリュームに書き込みをテスト

oc exec lvms-test-pod — sh -c "echo 'LVM Storage Test' > /data/test.txt && cat /data/test.txt"

期待される出力: LVM Storage Test

テストリソースをクリーンアップ:

[source,bash,role=execute]

oc delete pod lvms-test-pod oc delete pvc lvms-test-pvc

== forceWipeDevicesAndDestroyAllData の理解

=== 何をするのか

`forceWipeDevicesAndDestroyAllData: true` パラメーターは次のことを行います:

. **パーティションテーブルを削除**: GPT、MBR、およびパーティション情報を消去します
. **ファイルシステムを破壊**: ext4、xfs、btrfs、およびその他のファイルシステムの署名を削除します
. **LVM メタデータをクリア**: 既存の物理ボリューム、ボリュームグループ、および論理ボリュームを削除します
. **ディスクの先頭にゼロを書き込み**: 最初の数メガバイトにゼロを書き込み、クリーンな状態を確保します
. **再利用を強制**: LVM operator がディスクの所有権を取得できるようにします

=== 使用する場合

`forceWipeDevicesAndDestroyAllData: true` を次の場合に使用します:

* ディスクに既存のパーティションまたはファイルシステムがある
* ディスクが以前に LVM (PV/VG/LV を含む) で使用されていた
* ディスクに破棄したいデータがある
* ディスクを別の用途から再利用する

使用しない場合は:

* ディスクが新しく未形式化である
* 使用するディスクが不明である (まず確認してください!)
* ディスクに保持する必要があるデータがある

=== 使用前の安全チェック

常にディスクデバイスパスを確認する前に、`forceWipeDevicesAndDestroyAllData: true` パラメーターを有効にします:

[source,bash,role=execute]

1. ディスクパスを確認

oc debug node/<node-name> — chroot /host lsblk -o NAME,SIZE,TYPE,MOUNTPOINT

2. ディスクがマウントされていないことを確認

oc debug node/<node-name> — chroot /host mount | grep /dev/sda

3. システムディスクを使用していないことを確認

oc debug node/<node-name> — chroot /host df -h | grep sda

4. ディスクのシリアル番号を確認

oc debug node/<node-name> — chroot /host lsblk -d -o NAME,SERIAL /dev/sda

== 要約

このチュートリアルで学んだこと:

* クラスターノード上のスパアディスクを特定して LVM ストレージに使用する方法
* LVM Storage operator をインストールするために名前空間、OperatorGroup、およびサブスクリプションを作成する方法
* `forceWipeDevicesAndDestroyAllData` パラメーターを使用して既存データを強制的に消去するディスクの LVMCluster リソースを作成および構成する方法
* 自動生成される `lvms-vg1` StorageClass を OpenShift Virtualization のデフォルト StorageClass に設定する方法
* PVC と Pod を使用してストレージプロビジョニングをテストし、LVMCluster の健全性を確認する方法
