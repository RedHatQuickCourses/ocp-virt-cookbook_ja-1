VMライフサイクルステートと基本操作の理解

概要

OpenShift Virtualizationにおける仮想マシンのライフサイクルを理解することは、効果的なVM管理、トラブルシューティング、および自動化に不可欠です。このガイドでは、VirtualMachineリソースとVirtualMachineInstanceリソースの関係、VMが取りうるさまざまなステート、およびライフサイクル全体にわたるVMの管理方法について説明します。

テスト済みのバージョン:

OpenShift 4.20

前提条件

  • OpenShift VirtualizationがインストールされたOpenShift 4.18+

  • OpenShift WebコンソールまたはCLIツール (oc、virtctl) へのアクセス

  • クラスター内に少なくとも1つのVMが作成されていること (ウェブコンソールとテンプレートを使用したVMの作成を参照)

VirtualMachine vs VirtualMachineInstance

OpenShift Virtualizationは、仮想マシンを管理するために2つの主要なリソースを使用します。

VirtualMachine (VM)

`VirtualMachine`リソースは、VMの永続的な定義です。これは、以下のものを定義するテンプレートまたは仕様と考えることができます。

  • VMの名前とnamespace

  • CPUとメモリーの設定

  • ディスクとネットワークのアタッチメント

  • 起動順序とファームウェアの設定

  • 実行ストラテジー (VMがどのように動作すべきか)

VirtualMachineリソースは、VMが実行されているかどうかにかかわらず存在します。

# VirtualMachineリソースをリスト表示します
oc get vm -n my-vms

出力例:

NAME            AGE   STATUS    READY
fedora-app-vm   14m   Running   True

VirtualMachineInstance (VMI)

`VirtualMachineInstance`リソースは、VMの実際の実行中のインスタンスを表します。これはVMが起動したときに作成され、VMが停止したときに削除されます。VMIには以下が含まれます。

  • ランタイム情報 (IPアドレス、ノード配置)

  • 現在のフェーズ/ステート

  • ゲストエージェント情報 (インストールされている場合)

# VirtualMachineInstanceリソースをリスト表示します
oc get vmi -n my-vms

出力例:

NAME            AGE   PHASE     IP             NODENAME   READY
fedora-app-vm   78s   Running   10.128.0.184   ocplab01   True
停止中のVMはVirtualMachineリソースを持ちますが、VirtualMachineInstanceは持ちません。VMを起動するとVMIが作成され、停止するとVMIは削除されますが、VMリソースは残ります。

VMライフサイクルステート

VMはライフサイクル中にいくつかのステートを遷移します。これらのステートを理解することは、監視とトラブルシューティングに役立ちます。

ステートの説明

順序 ステート 説明 VMIは存在しますか？

1

Stopped

VMは定義されていますが、実行されていません。コンピューティングリソースは消費されません。

いいえ

2

Pending

VMIが作成され、前提条件 (ストレージ、ネットワーク) を待っています。

はい

3

Scheduling

Kubernetesスケジューラーが適切なノードを見つけています。

はい

4

Starting

VMが割り当てられたノードで起動中です。

はい

5

Running

VMが完全に動作しており、接続を受け付けています。

はい

6

Paused

VMの実行が一時停止されています。ステートはメモリーに保持されます。

はい

7

Stopping

グレースフルシャットダウンが進行中です。

はい

8

Terminating

VMIリソースの最終クリーンアップ。

はい (短時間)

VMライフサイクルの管理

VMの起動

停止中のVMを起動するには:

Webコンソール:

  1. 仮想化 → 仮想マシン に移動します。

  2. リストからVMを見つけます。

  3. VM名をクリックして詳細を開き、アクション → 開始 をクリックします。

  4. または、ケバブメニュー (3つの点) をクリックし、開始 を選択します。

CLI:

virtctl start fedora-app-vm -n my-vms

VMが起動したことを確認します:

# VMステータスを確認します
oc get vm fedora-app-vm -n my-vms

# VMIが作成されたことを確認します
oc get vmi fedora-app-vm -n my-vms

出力例:

NAME            AGE   STATUS    READY
fedora-app-vm   14m   Running   True

NAME            AGE   PHASE     IP             NODENAME   READY
fedora-app-vm   78s   Running   10.128.0.184   ocplab01   True

VMの停止

VMを停止すると、グレースフルシャットダウンが実行され、VMIが削除されます。

Webコンソール:

  1. VMの詳細ページに移動します。

  2. アクション → 停止 をクリックします。

CLI (グレースフルストップ):

virtctl stop fedora-app-vm -n my-vms

CLI (強制停止):

VMが応答しない場合は、強制停止を使用します。

virtctl stop fedora-app-vm -n my-vms --force
強制停止は、電源コードを抜くのと同等です。グレースフルシャットダウンが失敗した場合にのみ使用してください。

VMが停止したことを確認します:

# VMは停止ステータスを表示するはずです
oc get vm fedora-app-vm -n my-vms

# VMIは存在しなくなるはずです
oc get vmi fedora-app-vm -n my-vms

出力例:

NAME            AGE   STATUS    READY
fedora-app-vm   14m   Stopped   False

Error from server (NotFound): virtualmachineinstances.kubevirt.io "fedora-app-vm" not found

VMの一時停止と再開

一時停止は、VMの状態をメモリーに保持しながらVMの実行を凍結します。これは以下の場合に役立ちます。

  • 一時的にCPUリソースを解放する

  • 一貫性のあるスナップショットを取得する

  • デバッグまたは検査

一時停止されたVMは、ノード上で依然としてメモリーを消費します。CPU実行のみが停止されます。

Webコンソール:

  1. VMの詳細ページに移動します。

  2. アクション → 一時停止 をクリックします。

  3. 再開するには: アクション → 一時停止を解除 をクリックします。

CLI:

# VMを一時停止します
virtctl pause vm fedora-app-vm -n my-vms

# VMの一時停止を解除します
virtctl unpause vm fedora-app-vm -n my-vms

一時停止状態を確認します:

oc get vm fedora-app-vm -n my-vms

出力例:

NAME            AGE   STATUS   READY
fedora-app-vm   14m   Paused   False

VMの再起動

再起動は、停止後に起動を実行します。これは以下の場合に使用します。

  • 再起動が必要な設定変更を適用する場合

  • ゲストOSの問題から回復する場合

  • VMの状態をリフレッシュする場合

Webコンソール:

  1. VMの詳細ページに移動します。

  2. アクション → 再起動 をクリックします。

CLI:

virtctl restart fedora-app-vm -n my-vms
再起動は、グレースフルシャットダウンが完了するまで待ってから起動します。ゲストOSが応答しない場合、再起動に時間がかかったり、最初に強制停止が必要になったりする場合があります。

VMの削除

VMを削除する際、関連するストレージ (PVC/DataVolumes) も削除するかどうかを選択できます。

Webコンソール:

  1. VMの詳細ページに移動します。

  2. アクション → 削除 をクリックします。

  3. 確認ダイアログで、ディスクを削除するかどうかを選択します。

  4. 削除 をクリックします。

CLI (VMを削除し、ディスクは保持):

oc delete vm fedora-app-vm -n my-vms

デフォルトでは、VMが所有するDataVolumesは削除されます。それらを保持するには、まず所有者参照を削除するか、Webコンソールのオプションを使用します。

削除を確認します:

# VMは存在しないはずです
oc get vm fedora-app-vm -n my-vms

# PVCが保持されたかを確認します (必要に応じて)
oc get pvc -n my-vms

出力例:

Error from server (NotFound): virtualmachines.kubevirt.io "fedora-app-vm" not found

No resources found in my-vms namespace.
VMの削除は永続的です。削除する前に重要なデータのバックアップがあることを確認してください。

イベントは、VMで何が起こっているかについての詳細情報を提供します。

# 特定のVMのイベントを取得します
oc get events -n my-vms --field-selector involvedObject.name=fedora-app-vm

出力例:

リアルタイムでイベントを監視します:

oc get events -n my-vms -w

一般的なイベントとその意味:

イベント 説明

SuccessfulCreate

VMIが正常に作成されました

Started

VMが正常に起動しました

Stopped

VMが停止しました (グレースフルシャットダウン)

Killing

強制停止が開始されました

FailedScheduling

適切なノードが見つかりませんでした

FailedCreate

VMIの作成が失敗しました (詳細を確認してください)

実行ストラテジー

実行ストラテジーは、VMがライフサイクル全体でどのように自動的に動作するかを定義します。ストラテジーはVM仕様で設定され、VMが自動的に起動するか、停止後に再起動するか、停止したままになるかを決定します。

利用可能な実行ストラテジー

ストラテジー 動作 ユースケース

Always

VMは自動的に起動し、何らかの理由で停止した場合 (ゲストによるシャットダウンを含む) は再起動します。

常に実行されている必要がある本番ワークロード。

RerunOnFailure

VMは自動的に起動し、失敗した場合にのみ再起動します。ゲストによるシャットダウンではVMは停止したままになります。

クラッシュから回復すべきだが、意図的なシャットダウンを尊重するサービス。

Manual

VMは、`virtctl`またはWebコンソールを介して明示的に要求された場合にのみ起動または停止します。

開発/テストVM、または特定のスケジューリングニーズを持つワークロード。

Halted

VMは停止したままになります。起動コマンドは無視されます。

一時的に廃止されたVMまたはメンテナンスモード。

実行ストラテジーの設定

VM YAML内:

apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: my-vm
spec:
  runStrategy: Always    # または RerunOnFailure, Manual, Halted
  template:
    # ... VM template spec

Webコンソール経由:

  1. VMの詳細ページに移動します。

  2. YAML タブをクリックします。

  3. `spec.runStrategy`フィールドを修正します。

  4. 保存 をクリックします。

`runStrategy`と非推奨の`running`フィールドを同時に使用することはできません。新しいVMには`runStrategy`を使用してください。

実行ストラテジーと手動コマンド

`Always`または`RerunOnFailure`を使用する場合、以下の点に注意してください。

  • `virtctl stop`はVMを一時的に停止しますが、自動的に再起動します。

  • VMを停止したままにするには、実行ストラテジーを`Halted`または`Manual`に変更します。

  • ゲストによるシャットダウン (例: VM内部での`shutdown -h now`) は、ストラテジーによって動作が異なります。
