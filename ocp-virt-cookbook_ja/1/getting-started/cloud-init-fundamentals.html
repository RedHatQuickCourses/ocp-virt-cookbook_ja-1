OpenShift Virtualization VMのためのCloud-initの基本

概要

Cloud-initは、仮想マシンの初回起動時の初期化を自動化するための業界標準ツールです。 OpenShift Virtualizationでは、ほとんどすべてのVMがcloud-initを使用して、ユーザーの構成、パスワードの設定、SSHキーの挿入、パッケージのインストール、ネットワークアドレスの割り当てを行います。 このチュートリアルでは、cloud-initの仕組み、KubeVirtがVMにcloud-initをどのように提供するか、そしてVirtualMachineマニフェストにcloud-init設定を埋め込む方法を紹介します。

学習内容

  • cloud-initとは何か、なぜVMの自動化に重要なのか

  • 3つのcloud-initデータ型: userData、networkData、metadata

  • KubeVirtがNoCloudデータソースを介してVMにcloud-initデータを配信する方法

  • ユーザー作成、パッケージインストール、スクリプト実行のための一般的なcloud-configモジュール

  • VirtualMachineマニフェストにcloud-initを埋め込む2つのパターン (インライン vs Secret参照)

  • cloud-initが正常に実行されたことを確認する方法

テスト済みのバージョン:

OpenShift 4.18+

前提条件

  • OpenShift VirtualizationがインストールされたOpenShift 4.18+

  • oc および virtctl CLIツールがインストールされていること (VM管理のためのvirtctlの使用を参照)

  • YAML構文の基本的な理解

  • VMの作成に慣れていること (ウェブコンソールとテンプレートを使用したVMの作成を参照)

Cloud-initとは？

Cloud-initは、仮想マシンの初回起動時に内部で実行されるオープンソースフレームワークです。 外部ソースから設定データを読み取り、それを使用して、手動での設定が必要となるタスク (ユーザーアカウントの作成、SSHアクセスの設定、パッケージのインストール、設定ファイルの書き込み、スクリプトの実行など) を自動化します。

Cloud-initは、Fedora、RHEL、CentOS、Ubuntu、Debianを含むほとんどのLinuxディストリビューションでサポートされています。 主要なクラウドプラットフォーム (AWS、Azure、GCP) および、OpenShift Virtualizationを支えるKubeVirtを含む仮想化プラットフォーム全体で標準的な初期化メカニズムです。

cloud-initは、デフォルトではVMの初回起動時にのみ実行されます。インスタンスデータが明示的にクリアされない限り、その後の再起動ではcloud-initはスキップされます。

KubeVirtがCloud-initを配信する方法

従来のクラウド環境では、VMはメタデータサービス (HTTPエンドポイント 169.254.169.254) からcloud-initデータを取得します。 KubeVirtは、*NoCloudデータソース*と呼ばれる異なるアプローチを使用します。

NoCloudを使用する場合、KubeVirtは以下のことを行います。

  1. VirtualMachineマニフェスト (または参照されているSecret) からcloud-initデータを読み取ります

  2. データをISOイメージとしてフォーマットされた仮想ディスクにパッケージ化します

  3. virtioデバイスを介してディスクをVMにアタッチします

  4. ゲストOS内のCloud-initがディスクを検出し、データを読み取り、設定を適用します

これにより、cloud-initの設定はKubernetesリソース内に自己完結します。外部メタデータサービスは必要ありません。

spec:
  template:
    spec:
      volumes:
        - name: cloudinitdisk
          cloudInitNoCloud:        (1)
            userData: |
              #cloud-config
              ssh_pwauth: true
              user:
                name: fedora
                lock_passwd: false
                plain_text_passwd: changeme
1 cloudInitNoCloud ボリュームタイプは、KubeVirtにNoCloudデータソースディスクを作成するように指示します。

Cloud-initデータ型

Cloud-initは3種類の構成データを受け入れます。KubeVirtでは、これらは`cloudInitNoCloud`ボリュームの下のフィールドにマッピングされます。

userData (cloud-config)

主要な設定チャネルです。userDataには、`#cloud-config`ヘッダーで始まるYAMLファイルである*cloud-config*ドキュメントが含まれています。ここで、ユーザー、パッケージ、スクリプト、ファイル、およびほとんどのVMカスタマイズを定義します。

cloudInitNoCloud:
  userData: |
    #cloud-config
    ssh_pwauth: true
    user:
      name: fedora
      lock_passwd: false
      plain_text_passwd: s3cur3p4ss
      ssh_authorized_keys:
        - ssh-ed25519 AAAA... user@workstation
    packages:
      - vim
      - tmux
最初の行の`#cloud-config`ヘッダーは必須です。これがないと、cloud-initはそのデータをcloud-configドキュメントとして認識せず、処理をサイレントにスキップします。

networkData (network-config v2)

VMインターフェースのネットワーク設定。Netplan互換の*network-configバージョン2*形式を使用します。静的IP、カスタムDNS、またはマルチインターフェース設定が必要な場合に使用します。

シンプルなDHCP設定:

cloudInitNoCloud:
  networkData: |
    version: 2
    ethernets:
      enp1s0:
        dhcp4: true

DNSとルートを含む静的IP設定:

cloudInitNoCloud:
  networkData: |
    version: 2
    ethernets:
      enp1s0:
        addresses:
          - 192.168.1.100/24             (1)
        nameservers:
          addresses:
            - 192.168.1.1
            - 8.8.8.8                    (2)
        routes:
          - to: default
            via: 192.168.1.1             (3)
          - to: 10.0.0.0/8
            via: 192.168.1.254           (4)
1 CIDR表記のサブネットマスクを持つ静的IPアドレス。
2 DNSリゾルバーアドレス。
3 ルートとして定義されたデフォルトゲートウェイ (`gateway4`は非推奨)。
4 特定のゲートウェイを介して`10.0.0.0/8`ネットワークに到達するための追加ルート。

インターフェース名が予測できない場合に役立つ、名前の代わりにMACアドレスでインターフェースを照合することもできます。

cloudInitNoCloud:
  networkData: |
    version: 2
    ethernets:
      primary:
        match:
          macaddress: "52:54:00:12:34:56"
        addresses:
          - 192.168.1.100/24
        routes:
          - to: default
            via: 192.168.1.1
KubeVirt VMでは、最初のインターフェースは通常`enp1s0`、2番目は`enp2s0`などとなります。これらの名前は、NetworkAttachmentDefinitionsの順序ではなく、VirtualMachineスペックでインターフェースが定義されている順序に対応しています。

マルチインターフェース設定、VLAN、および高度なシナリオについては、静的IP構成を参照してください。

metadata

VMインスタンスのメタデータ (例: instance-id や local-hostname)。ほとんどの場合、KubeVirtはVirtualMachineリソースからメタデータを自動的に入力するため、手動で設定する必要はほとんどありません。

cloudInitNoCloud:
  metadata: |
    instance-id: my-vm-001
    local-hostname: my-vm
ホスト名を設定するだけでよい場合は、カスタムメタデータを提供する代わりにuserDataの`hostname`モジュールを使用してください。

Cloud-init実行ステージ

Cloud-initは、ブートプロセス中に5つのステージで実行されます。これらのステージを理解することで、設定がいつ有効になるかを予測するのに役立ちます。

ステージ 実行タイミング 内容

Generator

非常に早いブート時 (systemd generator)

cloud-initを実行するかどうかを決定する

Local

ネットワークが起動する前

networkDataからネットワーク構成を適用する

Network

ネットワークが利用可能になった後

残りのデータソース情報を取得し、ホスト名を設定する

Config

ネットワークステージの後

構成モジュールを実行する: ユーザー作成、ファイル書き込み、SSHキーインストール

Final

ブートの終盤

`runcmd`コマンドの実行、パッケージのインストール、最終スクリプトの実行

この順序付けは、ネットワーク設定 (networkDataから) がユーザー設定 (userDataモジュールから) より前に適用されることを意味します。`runcmd`で指定されたコマンドは、他のすべてのモジュールの後に最後に実行されます。

一般的なCloud-configモジュール

以下のモジュールは、最も一般的なVM初期化タスクをカバーします。各例は、#cloud-config userDataドキュメントの関連セクションを示しています。

ユーザーの作成とパスワードの設定

#cloud-config
ssh_pwauth: true
user:
  name: fedora
  lock_passwd: false
  plain_text_passwd: s3cur3p4ss
  ssh_authorized_keys:
    - ssh-ed25519 AAAA...key-data... user@workstation

`user`キーはデフォルトユーザーを設定します。`lock_passwd: false`と`plain_text_passwd`を設定して、パスワードログインを許可します。トップレベルに`ssh_pwauth: true`を追加して、SSHパスワード認証を有効にします (多くのクラウドイメージではデフォルトで無効になっています)。

デフォルト以外の追加ユーザーには、`users`リストを使用します。

#cloud-config
users:
  - name: admin
    groups: wheel
    sudo: ALL=(ALL) NOPASSWD:ALL
    ssh_authorized_keys:
      - ssh-ed25519 AAAA...key-data...
  - name: developer
    shell: /bin/bash

パッケージのインストール

#cloud-config
package_update: true
packages:
  - vim
  - git
  - tmux
  - python3
パッケージのインストールは*最終*ステージで実行されます。パッケージのインストールが完了する前でも、VMはコンソールを介してアクセス可能です。

コマンドの実行

#cloud-config
runcmd:
  - systemctl enable --now cockpit.socket
  - echo "VM initialized at $(date)" >> /var/log/vm-init.log

コマンドは*最終*ステージ中に順次実行されます。各エントリは文字列 (`/bin/sh`経由で実行) またはリスト (直接実行) のいずれかです。

ファイルの書き込み

#cloud-config
write_files:
  - path: /etc/motd
    content: |
      Welcome to the OpenShift Virtualization VM.
      Managed by cloud-init.
    permissions: '0644'
  - path: /etc/myapp/config.yaml
    content: |
      database: postgres
      port: 5432
    owner: root:root
    permissions: '0600'
`write_files`によって書き込まれるファイルは、`runcmd`が実行される前の*config*ステージで作成されます。これにより、設定ファイルを書き込んでから、`runcmd`でそれに依存するサービスを開始しても安全です。

VirtualMachineマニフェストへのCloud-initの埋め込み

VirtualMachineにcloud-initデータを提供するパターンは2つあります。

インラインのcloudInitNoCloud

最もシンプルなアプローチ — cloud-initデータをVirtualMachineマニフェストに直接埋め込みます。

minimal-cloud-init-vm.yamlをダウンロード

apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-cloud-init-demo
  namespace: demo
spec:
  runStrategy: Always
  template:
    metadata:
      labels:
        app: fedora-cloud-init-demo
    spec:
      domain:
        devices:
          disks:
            - disk:
                bus: virtio
              name: rootdisk
            - disk:
                bus: virtio
              name: cloudinitdisk
          interfaces:
            - masquerade: {}
              name: default
        resources:
          requests:
            memory: 2Gi
      networks:
        - name: default
          pod: {}
      volumes:
        - dataVolume:
            name: fedora-cloud-init-demo
          name: rootdisk
        - cloudInitNoCloud:                    (1)
            userData: |                        (2)
              #cloud-config
              ssh_pwauth: true
              user:
                name: fedora
                lock_passwd: false
                plain_text_passwd: s3cur3p4ss
                ssh_authorized_keys:
                  - 'ssh-ed25519 AAAA...your-key... user@workstation'
              users:
                - name: userone
                  lock_passwd: false
                  plain_text_passwd: s3cur3p4ss
                  ssh_authorized_keys:
                    - 'ssh-ed25519 AAAA...your-key... user@workstation'
              packages:
                - vim
                - git
              runcmd:
                - echo "Cloud-init complete" > /tmp/cloud-init-done
          name: cloudinitdisk
  dataVolumeTemplates:
    - metadata:
        name: fedora-cloud-init-demo
      spec:
        sourceRef:
          kind: DataSource
          name: fedora
          namespace: openshift-virtualization-os-images
        storage:
          resources:
            requests:
              storage: 30Gi
1 cloudInitNoCloud ボリュームタイプは、NoCloudデータソースディスクを作成します。
2 userData`フィールドには、#cloud-config`で始まるcloud-config YAMLが含まれます。

このパターンはシンプルな構成に適しています。より大規模な構成や機密性の高い構成の場合は、Secret参照を使用します。

Secret参照

複雑な構成の場合や、cloud-initデータに機密値 (パスワード、トークン) が含まれる場合は、データをKubernetes Secretに保存し、VirtualMachineから参照します。

cloud-init-secret-vm.yamlをダウンロード

まず、Secretを作成します。

oc create secret generic my-vm-cloud-init \
  --from-literal=userdata='#cloud-config
ssh_pwauth: true
user:
  name: fedora
  lock_passwd: false
  plain_text_passwd: s3cur3p4ss
  ssh_authorized_keys:
    - ssh-ed25519 AAAA...your-key... user@workstation
users:
  - name: userone
    lock_passwd: false
    plain_text_passwd: s3cur3p4ss
    ssh_authorized_keys:
      - ssh-ed25519 AAAA...your-key... user@workstation
packages:
  - vim
  - git
' \
  -n demo

次に、VirtualMachineマニフェストで参照します。

volumes:
  - cloudInitNoCloud:
      secretRef:
        name: my-vm-cloud-init         (1)
    name: cloudinitdisk
1 KubeVirtはSecretから`userdata`キーを読み取り、それをcloud-init userDataとして配信します。

Cloud-initが実行されたことを確認する

VMが起動した後、cloud-initが構成を正常に処理したことを確認します。

VMコンソールにアクセスします。

virtctl console fedora-cloud-init-demo -n demo

cloud-initのステータスを確認します。

cloud-init status --long

正常に実行された場合の期待される出力:

status: done
extended_status: done
boot_status_code: enabled-by-generator
detail:
DataSourceNoCloud [seed=/dev/vdb]

cloud-initがエラーを報告する場合は、ログを確認してください。

cat /var/log/cloud-init.log | tail -50

詳細な診断手順と一般的な問題の解決策については、Cloud-initのトラブルシューティングを参照してください。

よくある間違い

`#cloud-config`ヘッダーの欠落

userDataの最初の行は、正確に`#cloud-config`である必要があります。これがないと、cloud-initはuserDataブロック全体をエラーメッセージなしで無視します。

YAMLインデントエラー

Cloud-configはYAMLです。一貫性のないインデント (タブとスペースの混用、誤ったネストの深さ) は、解析の失敗を引き起こします。VM内で`cloud-init devel schema --config-file /path/to/config.yaml`を使用して構文を検証します。

ネットワーク設定をuserDataに配置する

ネットワークインターフェース設定は`userData`ではなく`networkData`に属します。network-config v2 YAMLをuserData内に配置しても効果はありません。

再起動時にcloud-initが再実行されることを期待する

Cloud-initは初回起動時に一度実行されます。再実行するには、`cloud-init clean`を実行してから再起動することで、その状態をクリアする必要があります。

Secretキー名の誤り

secretRef`を使用する場合、Secretには`userdata (すべて小文字) という名前のキーが含まれている必要があります。`userData`や`user-data`という名前のキーは見つかりません。

Windows VM: Sysprep

Cloud-initはLinux固有のツールです。Windows VMの場合、OpenShift Virtualizationは*Sysprep*をサポートしています。Sysprepは、Microsoft System Preparationツールであり、同様の初回起動時カスタマイズ (ホスト名、管理者パスワード、ドメイン参加、無人セットアップ) を実行します。

Windows VMの構成については、Windows VMへのリモートアクセスを参照してください。

クリーンアップ

このチュートリアルで作成したデモリソースを削除します。

oc delete vm fedora-cloud-init-demo -n demo
oc delete vm fedora-cloud-init-secret-demo -n demo
oc delete secret my-vm-cloud-init -n demo
oc delete namespace demo

まとめ

このチュートリアルでは、以下のことを学習しました。

  • Cloud-initは、設定を自動化するために初回起動時に実行される標準的なLinux VM初期化フレームワークである

  • KubeVirtは、cloud-initデータをNoCloudデータソースを介して配信し、VMにアタッチされたvirtioディスクとしてパッケージ化する

  • 3つのデータ型 — userData (ユーザー、パッケージ、スクリプト用のcloud-config)、networkData (network-config v2)、およびmetadata (インスタンスID)

  • Cloud-initは段階的に実行される: まずネットワーク構成が適用され、次にユーザー/ファイル構成、そしてコマンドとパッケージが続く

  • 2つのマニフェストパターン: シンプルなケースにはインラインの`cloudInitNoCloud`、複雑または機密性の高いデータにはSecret参照

  • cloud-init status --long`と/var/log/cloud-init.log`を使用した検証
