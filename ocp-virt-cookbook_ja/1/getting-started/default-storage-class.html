デフォルトストレージクラスの設定

概要

このチュートリアルでは、OpenShift Virtualization のデフォルトストレージクラスを設定する方法を説明します。デフォルトストレージクラスは、組み込みの DataSource イメージ (Fedora、RHEL、CentOS Stream、Windows) からブートソースを自動的にプロビジョニングするために必要です。これがないと、これらの DataSource は保留状態のままであり、VM を作成できません。デフォルトを設定することで、明示的にストレージが定義されていない場合に、仮想マシンとデータボリュームが指定されたストレージを自動的に使用するようになり、デプロイメントと管理が簡素化されます。

テスト済みバージョン:

OpenShift 4.19, 4.20

前提条件

開始するには、次のものが必要です。

  • OpenShift クラスター: OpenShift Virtualization がインストールされている OpenShift クラスター

  • クラスター管理者アクセス: クラスター管理者権限を持つクラスターへのアクセス

  • ストレージクラス: クラスターに少なくとも1つの StorageClass が設定されていること

  • ストレージプロビジョナー: 動的プロビジョニングをサポートするストレージプロビジョナー

ステップ 1: 利用可能なストレージクラスの確認

まず、クラスターで利用可能なストレージクラスを確認し、すでにデフォルトストレージクラスが設定されているかどうかを特定します。

oc get sc

このコマンドは、すべてのストレージクラスを名前とプロビジョナーとともに一覧表示します。名前の横に (default) と表示されているストレージクラスを探してください。これは、現在デフォルトとして設定されていることを示します。

デフォルトストレージクラスがない場合の出力例:

NAME                    PROVISIONER                             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
ceph-rbd-sc             rbd.csi.ceph.com                        Delete          Immediate              true                   2d
hostpath-provisioner    kubevirt.io/hostpath-provisioner        Delete          WaitForFirstConsumer   false                  1d
local-storage           kubernetes.io/no-provisioner            Delete          WaitForFirstConsumer   false                  1d

既存のデフォルトストレージクラスがある場合の出力例:

NAME                    PROVISIONER                             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
ceph-rbd-sc (default)   rbd.csi.ceph.com                        Delete          Immediate              true                   2d
hostpath-provisioner    kubevirt.io/hostpath-provisioner        Delete          WaitForFirstConsumer   false                  1d
local-storage           kubernetes.io/no-provisioner            Delete          WaitForFirstConsumer   false                  1d

次の点に注意してください。

  1. デフォルトとして設定したいストレージクラスの名前

  2. すでにデフォルトストレージクラスが設定されているかどうか

ステップ 2: 既存のデフォルトの削除 (必要な場合)

別のストレージクラスがすでにデフォルトとして設定されている場合、新しいデフォルトを設定する前に、そのアノテーションを最初に削除する必要があります。クラスターには1つのデフォルトストレージクラスしか設定できません。

ストレージクラスからデフォルトのアノテーションを削除するには:

oc annotate sc <the-old-default-storage-class-name> storageclass.kubernetes.io/is-default-class-
アノテーションキーの末尾のハイフンは、アノテーションを削除するものです。

ステップ 3: ストレージクラスへのアノテーション追加

選択したストレージクラスにアノテーションを追加します。このアノテーションは、OpenShift に対して、この特定のストレージクラスがデフォルトであるべきであることを伝えます。

アノテーションは次のとおりです: storageclass.kubernetes.io/is-default-class: "true"

oc annotate sc <your-storage-class-name> storageclass.kubernetes.io/is-default-class="true"

たとえば、ストレージクラスの名前が ceph-rbd-sc の場合:

oc annotate sc ceph-rbd-sc storageclass.kubernetes.io/is-default-class="true"

ステップ 4: デフォルトストレージクラスの確認

アノテーションを適用した後、変更が有効になっていることを確認します。

oc get sc

デフォルト設定後の出力例:

NAME                    PROVISIONER                             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
ceph-rbd-sc (default)   rbd.csi.ceph.com                        Delete          Immediate              true                   2d
hostpath-provisioner    kubevirt.io/hostpath-provisioner        Delete          WaitForFirstConsumer   false                  1d
local-storage           kubernetes.io/no-provisioner            Delete          WaitForFirstConsumer   false                  1d

選択したストレージクラス名の横に (default) と表示されるはずです。

ステップ 5: デフォルトストレージクラスのテスト

デフォルトストレージクラスが正しく機能していることを確認するための簡単なテストを作成します。ストレージクラスを指定せずに PVC を作成します。

oc apply -f - <<EOF
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: test-default-storage
  namespace: default
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
EOF

次に、PVC が正しいストレージクラスで作成されたことを確認します。

oc get pvc test-default-storage -n default

出力例:

NAME                   STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   AGE
test-default-storage   Pending                                      lvms-vg1       0s

PVC は Pending ステータスですが、デフォルトストレージクラス (この例では lvms-vg1) に自動的に割り当てられました。

ボリュームバインディングモードの理解

ほとんどのストレージクラスは WaitForFirstConsumer ボリュームバインディングモードを使用するため、PVC は Pending ステータスのままになります。これは期待される動作です。

Binding Mode 動作 ユースケース

WaitForFirstConsumer

PVC を使用する Pod がスケジュールされたときにのみ PV がプロビジョニングされます

Pod と同じノードにストレージが作成されることを保証します。トポロジーを認識するストレージ (LVM、ローカルストレージ) に適しています

Immediate

PVC が作成されるとすぐに PV がプロビジョニングされます

任意のノードからアクセス可能なネットワークアタッチストレージ (NFS、Ceph、クラウドストレージ) に適しています

ストレージクラスのバインディングモードを確認するには:

oc get sc <storage-class-name> -o jsonpath='{.volumeBindingMode}'

ステップ 6: デフォルトストレージを使用した VM のデプロイ

これで、データボリュームにデフォルトストレージクラスを自動的に使用する仮想マシンをデプロイできます。

oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: test-vm-default-storage
  namespace: default
spec:
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: test-vm-disk
    spec:
      pvc:
        accessModes:
        - ReadWriteOnce
        resources:
          requests:
            storage: 10Gi
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
  template:
    metadata:
      labels:
        kubevirt.io/vm: test-vm-default-storage
    spec:
      domain:
        devices:
          disks:
          - name: datavolumedisk
            disk:
              bus: virtio
        resources:
          requests:
            memory: 1Gi
            cpu: 1
      networks:
      - name: default
        pod: {}
      volumes:
      - name: datavolumedisk
        dataVolume:
          name: test-vm-disk
EOF

トラブルシューティング

よくある問題

複数のデフォルトストレージクラス

複数のストレージクラスがデフォルトとしてマークされている場合は、1つを除くすべてのアノテーションを削除します。

oc get sc -o custom-columns=NAME:.metadata.name,DEFAULT:.metadata.annotations.storageclass\.kubernetes\.io/is-default-class

ストレージクラスが見つからない

アノテーションを適用する前に、ストレージクラスが存在することを確認してください。

oc get sc <storage-class-name>

権限拒否

クラスター管理者権限があることを確認してください。

oc auth can-i create storageclass --all-namespaces

確認コマンド

すべてのストレージクラスとそのデフォルトステータスを確認します。

oc get sc -o wide

特定のストレージクラスに関する詳細情報を表示します。

oc describe sc <storage-class-name>

PVC とそのストレージクラスを一覧表示します。

oc get pvc --all-namespaces -o custom-columns=NAMESPACE:.metadata.namespace,NAME:.metadata.name,STORAGECLASS:.spec.storageClassName

ベストプラクティス

  1. 適切なストレージの選択: VM に必要なパフォーマンスと機能 (例: 高速 I/O、スナップショット、暗号化) を提供するストレージクラスを選択します。

  2. ワークロード要件の考慮: 異なるワークロードでは、異なるストレージ特性が必要になる場合があります。デフォルトに依存するのではなく、重要なワークロードには特定のストレージクラスを使用することを検討してください。

  3. ストレージ使用状況の監視: デフォルトストレージクラスがニーズを満たしていることを確認するために、ストレージの使用状況とパフォーマンスを定期的に監視します。

  4. 選択内容の文書化: 将来の参照とトラブルシューティングのために、特定のストレージクラスをデフォルトとして選択した理由を文書化します。

まとめ

このチュートリアルでは、次のことを学びました。

  • 利用可能なストレージクラスを表示し、既存のデフォルトを特定する方法

  • 既存のデフォルトストレージクラスアノテーションを削除する方法

  • storageclass.kubernetes.io/is-default-class を使用してストレージクラスをデフォルトとしてアノテーションを付ける方法

  • テスト PVC を作成し、VM をデプロイしてデフォルトストレージクラスを確認する方法

  • OpenShift Virtualization でデフォルトストレージクラスを選択および管理するためのベストプラクティス
