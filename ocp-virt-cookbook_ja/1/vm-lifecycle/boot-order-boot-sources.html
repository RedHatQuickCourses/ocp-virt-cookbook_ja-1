VM ブート順序とブートソースの構成

概要

このチュートリアルでは、OpenShift Virtualization で VM のブート順序、ブートソース、およびブートオプションを構成する方法を示します。ブート構成の理解は、カスタム OS インストール、リカバリー・シナリオ、マルチブート環境、およびセキュリティーコンプライアンス要件をサポートするために不可欠です。

ブート構成により、VM がブートを試みるデバイスの順序を制御し、使用するファームウェアモード (BIOS または UEFI) を指定できます。OpenShift Virtualization は、永続ボリューム、コンテナーイメージ、およびネットワークベースのソースなど、さまざまなブートソースをサポートしています。

前提条件

  • OpenShift 4.18 以上に OpenShift Virtualization オペレーターがインストールされている

  • CLI ツール: oc, virtctl, kubectl

  • VM ブートプロセスの基本的な理解

  • 永続ボリューム用のストレージクラスが設定されている

  • コンテナーレジストリーの例 (Quay.io、Docker Hub、または内部レジストリー) へのアクセス権限 (コンテナーレジストリーの例)

OpenShift Virtualization でのブートソースタイプ

OpenShift Virtualization は、以下のブートソースタイプをサポートしています。

ブートソースタイプ 説明 使用例

PersistentVolumeClaim (PVC)

永続ディスクに基づいてブートする

生産 VM、状態フルワークロード、カスタム OS インストール

ContainerDisk

コンテナー形式でパッケージされたディスクイメージからブートする

エピシーマル VM、テスト、CI/CD、不変性ワークロード

DataVolume

Containerized Data Importer (CDI) を使用してプロビジョニングされたディスクからブートする

HTTP、S3、またはレジストリー源からのイメージのインポート

CloudInitNoCloud

ブートディスクと共に頻繁に使用される初期化ディスク (ブート可能ではない)

VM の構成と初期化

ファームウェアモード

OpenShift Virtualization は、以下の 2 つのファームウェアモードをサポートしています。

BIOS (レガシーモード)

  • 伝統的な PC BIOS ファームウェア

  • オルダーのオペレーティングシステムと互換性がある

  • ブートデバイスのサイズが制限されている (MBR による場合、2TB)

  • 起動時間が短くなる

  • ほとんどの Linux ディストリビューションのデフォルト

UEFI (統一可拡張ファームウェアインターフェース)

  • モダンなファームウェアインターフェース

  • Secure Boot を必要とする

  • GPT パーティションテーブルと大きなディスクをサポートする

  • Windows 11 と現代の Windows Server バージョンに必要

  • セキュリティー機能が向上する

ブートデバイスの順序の理解

ブートデバイスの順序は、VM ファームウェアがブート可能なコンテンツを検索するデバイスの順序を決定します。KubeVirt で、ブート順序はディスクおよびインターフェースの bootOrder フィールドを使用して制御されます。

重要な概念:

  • bootOrder の数値が小さいほど優先度が高くなる (1 が 2 よりも先にブートする)

  • ブート順序を指定できるのはディスクおよびインターフェースのみ

  • bootOrder が指定されていない場合、ファームウェアが決定する

  • 通常、主ブートディスクに bootOrder: 1 を割り当てる

シナリオ 1: PersistentVolumeClaim (最も一般的な)

これは、永続ストレージを使用する生産 VM の標準構成です。

テスト用の名前空間を作成します:

oc create namespace boot-demo

警告: この例のクラウド-init パスワードは、デモのためのものです。実際の生産環境では、強力で一意のパスワードまたは SSH キー認証を使用してください。
