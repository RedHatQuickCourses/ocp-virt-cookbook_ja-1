ユーザー定義ネットワーク (UDN) を使用したレイヤー 2 プライマリーネットワークの構成

概要

このチュートリアルでは、OpenShift Virtualization でレイヤー 2 トポロジーを使用したユーザー定義ネットワーク (UDN) を使用してプライマリーネットワークを作成する方法を示します。クラスターのデフォルトネットワークを使用せずに、UDN はカスタム IP 範囲を定義し、特定の名前空間内のアプリケーションに完全なネットワーク隔離を提供します。このアプローチは、アプリケーションが IP アドレスの直接制御を必要とするか、カスタムネットワーキング要件を持つ場合に適しています。これらの設定は、単純な 5 つのステップで完了し、追加のネットワーク接続定義は必要ありません。

テスト済みバージョン:

OpenShift 4.19, 4.20, 4.21

前提条件

  • OpenShift 4.19+ クラスターに OpenShift Virtualization オペレーターがインストールされている

  • OVN-Kubernetes ネットワークプラグイン (OpenShift のデフォルト)

  • 名前空間と UserDefinedNetwork リソースを作成するためのクラスター管理者権限

  • CLI ツールがインストールされている: oc, virtctl

ステップ 1: UDN プライマリーネットワークを有効にする名前空間の作成

UDN プライマリーネットワーク機能を有効にするために、UDN プライマリーネットワークラベルを持つ名前空間を作成します。特別なラベル k8s.ovn.org/primary-user-defined-network は、OpenShift に、この名前空間がクラスターのデフォルトネットワークではなくカスタムプライマリーネットワークを使用することを示します。

注: ワークロードのデプロイが行われる前に名前空間を作成してラベル付けする必要があり、作成後は変更できません。

udn-primary-namespace.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: udn-primary-vm-guests
  labels:
    k8s.ovn.org/primary-user-defined-network: ""
EOF

ステップ 2: UserDefinedNetwork リソースの構成

レイヤー 2 トポロジーとカスタムサブネット構成を持つプライマリー UDN を定義します。このリソースは、ネットワークの特性を指定します。これには IP 範囲、IPAM ライフサイクル、およびポッドのプライマリーロールの指定が含まれます。これにより、デフォルトクラスターネットワーキングがオーバーライドされ、名前空間内のポッドにカスタム IP アドレスが割り当てられます。

udn-primary.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: k8s.ovn.org/v1
kind: UserDefinedNetwork
metadata:
  name: udn-primary
  namespace: udn-primary-vm-guests
spec:
  topology: Layer2
  layer2:
    role: Primary
    subnets:
    - "192.168.100.0/24"
  ipam:
    lifecycle: Persistent
EOF

ステップ 3: UDN を使用したワークロードのデプロイ

192.168.100.0/24 サブネットから IP アドレスを自動的に取得し、UDN の IPAM を使用して UDN をプライマリーネットワークインターフェースとして使用する仮想マシンを作成します。この仮想マシンは、単純な Python HTTP サーバーを使用して接続性をテストします。

警告: この例では、cloud-init パスワードはデモ目的のみです。実際の生産環境では、強力で一意のパスワードまたは SSH キー認証を使用してください。

fedora-vm-with-udn.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-vm-with-udn
  namespace: udn-primary-vm-guests
  labels:
    app: fedora-web-vm
    network-type: udn-primary
spec:
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: fedora-udn-volume
    spec:
      pvc:
        accessModes:
        - ReadWriteOnce
        resources:
          requests:
            storage: 33Gi
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
  template:
    metadata:
      labels:
        app: fedora-web-vm
    spec:
      domain:
        devices:
          disks:
          - name: datavolumedisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            binding:
              name: l2bridge
        resources:
          requests:
            memory: 2Gi
            cpu: 1
      networks:
      - name: default
        pod: {}
      volumes:
      - name: datavolumedisk
        dataVolume:
          name: fedora-udn-volume
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            write_files:
            - path: /etc/resolv.conf
              content: |
                nameserver 172.30.0.10      # クラスター DNS
                search udn-primary-vm-guests.svc.cluster.local svc.cluster.local
                options ndots:5
            user: fedora
            password: fedora
            chpasswd: { expire: False }
            packages:
            - python3
            runcmd:
            - echo "<h1>Welcome to OpenShift Virtualization !!!</h1>" > /root/index.html
            - cd /root && nohup python3 -m http.server 80 > /dev/null 2>&1 &
EOF

ステップ 4: UDN プライマリーネットワークを持つ仮想マシンをサービスとして公開

VM の Python HTTP サーバーに LoadBalancer を使用してアクセスを公開する Kubernetes Service を作成します。

注: AWS などのクラウドサービスは、この構成に自動的にロードバランサーをデプロイします。裸機デプロイメントの場合は、MetallB をロードバランサーとして考慮してください。

fedora-vm-web-service.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: v1
kind: Service
metadata:
  name: fedora-vm-web-service
  namespace: udn-primary-vm-guests
  labels:
    app: fedora-web-vm
spec:
  selector:
    app: fedora-web-vm
  ports:
  - name: http
    port: 80
    targetPort: 80
    protocol: TCP
  type: LoadBalancer
EOF

ステップ 5: サービスの可用性を確認

AWS ベースのクラスターに基づくメタルタイプインスタンスを使用している場合、ロードバランサーが自動的にデプロイされ、外部名が作成されます。外部サービス IP で名前を見つけることができます:

oc get svc

例出力:

NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) AGE fedora-vm-web-service LoadBalancer 172.30.72.140 aa32981df19ad4959ae9a82fb9990db9-1996929606.ca-central-1.elb.amazonaws.com 80:32291/TCP 45m

curl を使用してサービスをテストします:

[source,bash,role=execute]

curl aa32981df19ad4959ae9a82fb9990db9-1996929606.ca-central-1.elb.amazonaws.com

期待される出力:

<h1>Welcome to OpenShift Virtualization !!!</h1>

== 要約

このチュートリアルで学んだこと:

* `k8s.ovn.org/primary-user-defined-network` ラベルを持つ名前空間を作成する方法
* Layer2 トポロジーとカスタムサブネット構成を持つ `UserDefinedNetwork` リソースを定義する方法
* UDN の IPAM を持つ持続的な IP アドレス割り当てを持つ仮想マシンをデプロイする方法
* UDN プライマリーネットワークを持つ仮想マシンを LoadBalancer Service で公開する方法
* UDN プライマリーネットワークが完全なネットワーク隔離とカスタム IP アドレスを提供する方法
