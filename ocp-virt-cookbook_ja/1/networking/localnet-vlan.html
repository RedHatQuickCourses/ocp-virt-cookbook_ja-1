ローカルネットワークのセカンダリーネットワークの VLAN 設定に NetworkAttachmentDefinition を使用した OpenShift Virtualization の構成

このチュートリアルでは、NetworkAttachmentDefinition (NAD) を使用してローカルネットワークのセカンダリーネットワークに VLAN タグを付けて構成する方法を示します。このアプローチにより、直接的な物理ネットワークインフラストラクチャへのアクセスと VLAN 分割が可能になり、仮想マシン (VM) は特定の VLAN タグ付きネットワークと通信できるようになり、ポッドネットワーク接続を維持できます。これは、既存の VLAN ベースのネットワークインフラストラクチャとの統合が必要な場合や、ネットワーク分割が必要な場合に適しています。

前提条件

  • NMState Operator: クラスター内でインストールおよび実行されている必要があります

  • NMState CR: オペレーターが実行されている後に nmstate インスタンスを作成する必要があります

  • ワーカーノード: ブリッジマッピングはワーカーノードに適用されます

  • VLAN インフラストラクチャ: 物理ネットワークインフラストラクチャが VLAN タグ付けをサポートする必要があります

  • スイッチ構成: OpenShift ワーカーノードに接続されている物理スイッチポートは、複数の VLAN タグ付きトラフィックストリームを処理するために トランク モード で構成する必要があります

テスト済みバージョン:

OpenShift 4.19

ネットワーク構成要件

物理スイッチ構成: トランク モード

OpenShift ワーカーノードに接続されている物理スイッチポートは、ローカルネットワーク VLAN 設定が正しく動作するために トランク モード で構成する必要があります:

  • トランク モード: スイッチポートが複数の VLAN のトラフィックを運ぶことを許可します

  • VLAN タグ付け: 各 VLAN は一意の VLAN ID (例: 100、200、300) で識別されます

  • 複数のネットワーク: 複数の VLAN ベースのセカンダリーネットワークをサポートするために不可欠です

ステップ 1: VM ネームスペースの作成

ローカルネットワーク VLAN 設定を使用する VM に対してネームスペースを作成します。

localnet-vlan-namespace.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: vm-guests-vlan
EOF

ステップ 2: ブリッジマッピングの構成

logical network interface name localnet-vlan を br-ex ブリッジインターフェースに接続するブリッジマッピングを構成します。このマッピングは、OVN-Kubernetes がローカルネットワーク VLAN サポートを処理する物理ブリッジインターフェースを知らせ、VM が VLAN タグ付き外部ネットワークにアクセスできるようにします。

localnet-vlan-bridge-mapping-nncp.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: localnet-vlan-bridge-mapping
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ''
  desiredState:
    ovn:
      bridge-mappings:
      - localnet: localnet-vlan
        bridge: br-ex
        state: present
EOF

ステップ 3: VLAN を持つ NetworkAttachmentDefinition の作成

ローカルネットワークトポロジーに VLAN タグ付けを持つセカンダリーネットワーク接続を構成する NetworkAttachmentDefinition を作成します。

localnet-vlan-100-nad.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: "k8s.cni.cncf.io/v1"
kind: NetworkAttachmentDefinition
metadata:
  name: localnet-vlan-100
  namespace: vm-guests-vlan
spec:
  config: '{
    "cniVersion": "0.3.1",
    "name": "localnet-vlan-100",
    "type": "ovn-k8s-cni-overlay",
    "topology": "localnet",
    "netAttachDefName": "vm-guests-vlan/localnet-vlan-100",
    "physicalNetworkName": "localnet-vlan",
    "vlanID": 100,
    "subnets": "192.168.100.0/24",
    "excludeSubnets": "192.168.100.1/32"
  }'
EOF

構成詳細:

  • physicalNetworkName: "localnet-vlan": NNCP で構成されたブリッジマッピング (localnet 名に一致する必要があります) にこの NetworkAttachmentDefinition (NAD) をリンクします

  • vlanID: 100: ネットワーク分割に VLAN タグ 100 を指定します

  • subnets: OVN-Kubernetes IPAM における IP サブネット範囲を定義します

  • excludeSubnets: ゲートウェイ IP の割り当てを除外します (通常、ルーター/ゲートウェイとして予約されます)

  • topology: "localnet": 直接的な物理ネットワークアクセスを持つ localnet トポロジーを使用します

ステップ 4: VLAN ネットワーク接続を持つ VM のデプロイ

ポッドネットワークと VLAN タグ付きローカルネットワークの両方に接続する VM を作成します。

警告: この例の cloud-init パスワードはデモのためのものです。実際の環境では、強力で一意のパスワードまたは SSH キー認証を使用してください。

fedora-vlan-vm.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-vlan-vm
  namespace: vm-guests-vlan
  labels:
    app: fedora-vlan-vm
spec:
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: fedora-vlan-volume
    spec:
      pvc:
        accessModes:
        - ReadWriteOnce
        resources:
          requests:
            storage: 33Gi
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
  template:
    metadata:
      labels:
        app: fedora-vlan-vm
    spec:
      domain:
        devices:
          disks:
          - name: datavolumedisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            bridge: {}
          - name: vlan_network
            bridge: {}
        resources:
          requests:
            memory: 2Gi
            cpu: 1
      networks:
      - name: default
        pod: {}
      - name: vlan_network
        multus:
          networkName: localnet-vlan-100
      volumes:
      - name: datavolumedisk
        dataVolume:
          name: fedora-vlan-volume
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            user: fedora
            password: fedora
            chpasswd: { expire: False }
            packages:
            - python3
            - tcpdump
            - net-tools
            runcmd:
            - echo "<h1>OpenShift Virtualization VLAN ネットワークへようこそ！</h1>" > /root/index.html
            - echo "<p>VLAN ID: 100</p>" >> /root/index.html
            - echo "<p>ネットワーク: 192.168.100.0/24</p>" >> /root/index.html
            - cd /root && nohup python3 -m http.server 8080 > /dev/null 2>&1 &
EOF

ステップ 5: VLAN ネットワーク接続を確認

VM コンソールを使用して、VLAN 設定を持ったダブルネットワーク接続を確認します。

# VM の状態を確認
oc get vm -n vm-guests-vlan

# 実行中の VM インスタンスを確認
oc get vmi -n vm-guests-vlan

# コンソール (cloud-init による認証: fedora/fedora) にアクセス
virtctl console -n vm-guests-vlan fedora-vlan-vm

VM コンソール内で、VLAN ネットワーク構成を確認します:

# 主 (ポッドネットワーク) とセカンダリー (VLAN ネットワーク) のネットワークインターフェースを確認
ip addr show

# デフォルトルートを確認 (低いメトリックが優先される)
# 主ネットワーク: メトリック 100、セカンダリー VLAN: メトリック 101
ip route show

# 予想される出力例:
default via 10.128.0.1 dev enp1s0 proto dhcp src 10.128.0.172 metric 100
default via 192.168.100.1 dev enp2s0 proto dhcp src 192.168.100.50 metric 101

# 外部接続性をテスト
ping 8.8.8.8

# Web サービスが実行中であることを確認
curl localhost:8080

# VLAN インターフェース接続をテスト (vlan-ip を実際の IP に置き換え)
curl <vlan-ip>:8080

# tcpdump が利用可能な場合の VLAN タグ付けの確認 (if tcpdump が利用可能な場合)
# これは、セカンダリーインターフェースで VLAN タグ付きトラフィックを表示します
sudo tcpdump -i enp2s0 -nn vlan 100

トラブルシューティング

一般的な問題

  1. VLAN トラフィックが VM に到達しない:

    • 物理スイッチ構成が VLAN タグ付けをサポートしていることを確認します

    • ワーカーノード上のブリッジマッピング構成を確認します

    • ネットワークインフラストラクチャと VLAN ID が一致していることを確認します

  2. IP アドレス割り当ての問題:

    • VLAN ネットワークで DHCP サーバーが利用可能であることを確認します

  3. ネットワーク接続の問題:

    • 物理ネットワークが構成された VLAN をサポートしていることを確認します

    • 物理ネットワークから VLAN サブネットへの接続をテストします

確認コマンド

# NetworkAttachmentDefinition の状態を確認
oc get network-attachment-definitions -n vm-guests-vlan

# ワーカーノード上のブリッジマッピングを確認
oc get nncp localnet-vlan-bridge-mapping -o yaml

# VM ネットワーク状態を確認
oc get vmi -n vm-guests-vlan -o yaml | grep -A 10 networks

# OVN-Kubernetes ログを確認してトラブルシューティングを行う
oc logs -n openshift-ovn-kubernetes -l app=ovnkube-node

要約

このチュートリアルで学んだこと:

  • ワーカーノード上の NodeNetworkConfigurationPolicy で VLAN サポートを持つブリッジマッピングを構成する方法

  • localnet トポロジー、VLAN タグ付け、OVN-Kubernetes IPAM を持つ NetworkAttachmentDefinition を作成する方法

  • 物理ネットワークポートがトランク モードで構成されていることを確認して、VLAN タグ付きトラフィックをサポートする必要があること

  • ポッドネットワークと VLAN タグ付きローカルネットワークの両方に接続する VM をデプロイする方法

  • VLAN ネットワーク構成を確認し、一般的な VLAN 接続問題をトラブルシューティングする方法
