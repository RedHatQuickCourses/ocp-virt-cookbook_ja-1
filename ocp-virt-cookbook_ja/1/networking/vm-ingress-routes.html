OpenShift Ingress を使用したHTTP/HTTPSトラフィックのVMとの統合

概要

このチュートリアルでは、OpenShift Route と Service を使用して、OpenShift Virtualization VM 内で実行されているアプリケーションを外部クライアントに公開する方法を説明します。Webサーバーを実行するVMをデプロイし、内部でVMに到達するためのClusterIP Serviceを作成し、オプションでTLS終端を伴うOpenShift Route を介してVMを外部に公開します。

このアプローチでは、コンテナワークロードをサービスするのと同じOpenShift Ingress インフラストラクチャを使用し、VMとPodの両方に対して一貫した外部アクセスモデルを提供します。

前提条件

  • OpenShift 4.18+ と OpenShift Virtualization オペレーターがインストール済みであること

  • クラスター管理者アクセス (system:admin または同等)

  • oc CLI および virtctl CLI がインストール済みであること

  • デフォルトのStorageClass が設定済みであること

  • Ingressコントローラー が実行中であること (OpenShift のインストールでデフォルト)

テスト済みバージョン:

OpenShift 4.20

ステップ1: Namespace を作成する

oc create namespace ingress-demo

ステップ2: Webサーバーを含むVMをデプロイする

cloud-init を使用してNginx Webサーバーを自動的に起動するFedora VMをデプロイします。VMには、Service が選択できるようにラベルが付けられています。

この例のcloud-initパスワードはデモ目的のみです。本番環境では、常に強力でユニークなパスワードまたはSSHキー認証を使用してください。

vm-webserver.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: web-vm
  namespace: ingress-demo
spec:
  runStrategy: Always
  template:
    metadata:
      labels:
        kubevirt.io/domain: web-vm
        app: web-vm
    spec:
      domain:
        devices:
          disks:
          - name: rootdisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          interfaces:
          - name: default
            masquerade: {}
        resources:
          requests:
            memory: 1Gi
      networks:
      - name: default
        pod: {}
      volumes:
      - name: rootdisk
        containerDisk:
          image: quay.io/containerdisks/fedora:latest
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            #cloud-config
            password: redhat
            chpasswd:
              expire: false
            packages:
            - nginx
            write_files:
            - path: /usr/share/nginx/html/index.html
              content: |
                <html>
                <head><title>OpenShift Virt - Ingress Demo</title></head>
                <body>
                <h1>Hello from OpenShift Virtualization</h1>
                <p>This page is served by Nginx inside a VM, exposed via an OpenShift Route.</p>
                </body>
                </html>
            runcmd:
            - systemctl enable nginx
            - systemctl start nginx
EOF

spec.template.metadata.labels の app: web-vm ラベルはvirt-launcher pod に渡され、Service がそれを選択できるようになります。

VMが起動するのを待ちます。

oc get vm -n ingress-demo -w

VMが Running と Ready=True を表示したら、次のステップに進みます。Ctrl+C を押して監視を終了します。

ステップ3: Webサーバーを確認する

cloud-init がNginxのインストールと起動を完了するまで待ちます (VMがRunning状態になってから1〜2分かかる場合があります)。

VMコンソールに接続します。

virtctl console web-vm -n ingress-demo

ユーザー名 fedora とパスワード redhat でログインし、次のコマンドを実行します。

curl -s http://localhost

カスタムHTMLページが表示されるはずです。Ctrl+] を押してコンソールを終了します。

ステップ4: Service を作成する

VMのポート80をターゲットとするClusterIP Service を作成します。Service のセレクターは、VMテンプレートに定義されたラベルと一致する必要があります。

service-web-vm.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: v1
kind: Service
metadata:
  name: web-vm-svc
  namespace: ingress-demo
spec:
  selector:
    app: web-vm
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
  type: ClusterIP
EOF

Service が存在することを確認します。

oc get svc web-vm-svc -n ingress-demo

エンドポイントがあることを確認します。

oc get endpoints web-vm-svc -n ingress-demo

期待される出力:

NAME         TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)   AGE
web-vm-svc   ClusterIP   172.30.x.x      <none>        80/TCP    5s

NAME         ENDPOINTS          AGE
web-vm-svc   10.128.x.x:80      5s

エンドポイントIPがVMのPod IPと一致し、Service がVMに正しくルーティングされていることを確認できます。

クラスター内から内部接続をテストします。

oc run curl-test --rm -it --restart=Never --image=registry.access.redhat.com/ubi9/ubi:latest -n ingress-demo -- curl -s http://web-vm-svc.ingress-demo.svc.cluster.local

HTMLページの内容が表示されるはずです。

ステップ5: OpenShift Route を介してService を公開する

HTTP Route (自動生成された証明書によるエッジ終端)

エッジTLS終端でRouteを作成します。OpenShift はRouteホスト名用の証明書を自動的に提供します。

route-web-vm.yaml をダウンロード

oc apply -f - <<EOF
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: web-vm-route
  namespace: ingress-demo
spec:
  to:
    kind: Service
    name: web-vm-svc
    weight: 100
  port:
    targetPort: 80
  tls:
    termination: edge
    insecureEdgeTerminationPolicy: Redirect
  wildcardPolicy: None
EOF

設定パラメーター

  • spec.to.name: ステップ4で作成したServiceを参照します。

  • spec.port.targetPort: Serviceポート (80) と一致する必要があります。

  • tls.termination: edge: TLSはルーターで終端されます。ルーターからVMへのトラフィックはプレーンなHTTPです。

  • insecureEdgeTerminationPolicy: Redirect: HTTPリクエストは自動的にHTTPSにリダイレクトされます。

Route を確認する

oc get route web-vm-route -n ingress-demo

期待される出力:

NAME           HOST/PORT                                          PATH  SERVICES     PORT  TERMINATION     WILDCARD
web-vm-route   web-vm-route-ingress-demo.apps.ocplab01.example.com       web-vm-svc   80    edge/Redirect   None

外部アクセスをテストする

Routeホスト名を取得します。

ROUTE_HOST=$(oc get route web-vm-route -n ingress-demo -o jsonpath='{.spec.host}')
echo "Route URL: https://$ROUTE_HOST"

curl で外部アクセスをテストします。

curl -sk https://$ROUTE_HOST

VM内のNginxによって提供されるHTMLページが表示されるはずです。

ワークステーションからRouteホスト名のDNS解決ができない場合は、/etc/hosts にホスト名をクラスターIngress IPを指すエントリを追加するか、curl --resolve を使用してください。

ステップ6: パススルーTLS (オプション)

VM内で独自のTLS証明書を処理するアプリケーションの場合、パススルー終端を使用します。ルーターは暗号化されたトラフィックを復号化せずにVMに直接転送します。

oc apply -f - <<EOF
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: web-vm-passthrough
  namespace: ingress-demo
spec:
  to:
    kind: Service
    name: web-vm-svc
    weight: 100
  port:
    targetPort: 443
  tls:
    termination: passthrough
  wildcardPolicy: None
EOF
パススルーは、VM内のアプリケーションがターゲットポートでHTTPSをサービスしている必要があります。このチュートリアルのNginxインスタンスはHTTPのみをサービスするため、このRouteタイプは参照として示されています。これを使用するには、VM内でTLS証明書を使用してNginxを設定してください。

仕組み

OpenShift Route を介して公開されたVMのトラフィックフロー:

  1. 外部クライアントはRouteホスト名にHTTPSリクエストを送信します。

  2. OpenShift Ingress Controller (HAProxyルーター) がTLSを終端します (エッジ終端の場合)。

  3. ルーターはリクエストをClusterIP Serviceのポート80に転送します。

  4. Service はトラフィックをvirt-launcher podのIPのポート80にルーティングします。

  5. virt-launcher pod内で、リクエストはマスカレードネットワークインターフェースを介してVMのNginxプロセスに到達します。

これはコンテナワークロードが公開される方法と同一であり、VMはOpenShift ネットワーキングモデルのファーストクラスの市民となります。

トラブルシューティング

Route が「503 Service Unavailable」を返す場合

Service に正常なエンドポイントがありません。以下を確認してください。

oc get endpoints web-vm-svc -n ingress-demo

空の場合、Service セレクターがVMラベルと一致しているか確認してください。

oc get pods -n ingress-demo -l app=web-vm

Route が「504 Gateway Timeout」を返す場合

VMアプリケーションが期待されるポートで応答していません。VMコンソールに接続します。

virtctl console web-vm -n ingress-demo

VM内で、Nginxが実行中で応答しているか確認します。

systemctl status nginx
curl -s http://localhost

DNS解決が失敗する場合

Routeホスト名が解決されない場合、Ingress ワイルドカードDNSが設定されているか確認してください。

oc get ingresscontroller default -n openshift-ingress-operator -o jsonpath='{.status.domain}'

Routeホスト名は <route-name>-<namespace>.<ingress-domain> のパターンに従います。

クリーンアップ

このチュートリアルで作成したすべてのリソースを削除します。

oc delete namespace ingress-demo

まとめ

このチュートリアルでは、次のことを学習しました。

  • cloud-init を使用してWebサーバーを実行するVMをデプロイする方法

  • VMポートを内部で公開するためのKubernetes Service を作成する方法

  • 外部HTTPSアクセス用のエッジTLS終端を備えたOpenShift Route を作成する方法

  • 独自の証明書を管理するVMのパススルーTLSを理解する方法

  • VMに関する一般的なService と Route の問題をトラブルシューティングする方法
