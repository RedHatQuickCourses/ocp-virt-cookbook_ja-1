リモートアクセスのLinux VMのトラブルシューティング

このガイドでは、OpenShift VirtualizationでリモートアクセスのLinux VMに関連する問題のトラブルシューティング手順、検証コマンド、および解決策を提供します。

クイックヘルスチェック

VMの状態を確認する

oc get vm <vm-name> -n <namespace>
oc get vmi <vm-name> -n <namespace>

実行中のVMの期待出力:

NAME        AGE   STATUS    READY
my-vm       5m    Running   True

VMにIPアドレスがあるか確認する

oc get vmi <vm-name> -n <namespace> -o jsonpath='{.status.interfaces[*].ipAddress}'

少なくとも1つのIPアドレスが返されるはずです。

VMのPodが実行中か確認する

oc get pods -n <namespace> -l kubevirt.io/domain=<vm-name>

期待出力:

NAME                       READY   STATUS    RESTARTS   AGE
virt-launcher-<vm>-xxxxx   3/3     Running   0          5m

ゲストエージェントの状態を確認する

oc get vmi <vm-name> -n <namespace> -o jsonpath='{.status.guestOSInfo.id}'

空の場合、ゲストエージェントは実行されていません。注: 基本的なアクセス方法（コンソール、VNC、SSH、ポートフォワード、NodePort、LoadBalancer、第二のネットワーク）では、ゲストエージェントは必要ありません。ゲストOSの詳細情報を提供し、動的SSH鍵のインジェクションなどの機能を有効にするために、追加情報を提供します。

方法1: コンソールアクセス

問題: コンソールへの接続ができない

症状:

$ virtctl console my-vm -n my-namespace
Error: unable to connect to console

診断:

# VMが実行中か確認する
oc get vm my-vm -n my-namespace

# VMIが存在するか確認する
oc get vmi my-vm -n my-namespace

# virt-launcher Podを確認する
oc get pods -n my-namespace -l kubevirt.io/domain=my-vm

一般的な原因: - VMが停止中 - VMIが作成されていない - virt-launcher Podが実行されていない - クライアントとAPIサーバー間のネットワーク問題

解決策:

  1. 停止中のVMを開始する:

    virtctl start my-vm -n my-namespace

  2. VMIが準備ができているまで待つ:

    oc wait --for=condition=Ready vmi/my-vm -n my-namespace --timeout=60s

  3. virt-launcher Podのログを確認する:

    oc logs -n my-namespace -l kubevirt.io/domain=my-vm

  4. virtctlバージョンがクラスターと一致するか確認する:

    virtctl version
oc get kubevirt -n openshift-cnv -o jsonpath='{.items[0].status.observedKubeVirtVersion}'

問題: コンソールに接続されていますが、ログインプロンプトが表示されていない

症状: コンソールに接続できますが、ログインプロンプトが表示されず、または空の画面が表示されます。

診断:

# 複数回Enterキーを押す

一般的な原因: - VMがまだブート中 - VMにシリアルコンソールが構成されていない - ttyS0でsystemd-gettyが実行されていない

解決策:

  1. VMが完全にブートされるまで待つ (VMのフェーズを確認):

    oc get vmi my-vm -n my-namespace -o jsonpath='{.status.phase}'

  2. ゲストOSがシリアルコンソールを有効にしているか確認する:

    • RHEL/Fedora: systemctl status serial-getty@ttyS0.service

    • クラウドイメージでは通常、デフォルトで有効になっています

  3. 断線して再接続してみる:

    # Ctrl+]を押して断線
virtctl console my-vm -n my-namespace

問題: コンソールでログインできない

症状: ログインプロンプトが表示されますが、ユーザー名とパスワードが機能しません。

一般的な原因: - ユーザー名またはパスワードが間違っている - cloud-initのSSH-onlyアクセスが構成されている - ルートログインが無効になっている

解決策:

警告: この例では、cloud-initのパスワードはデモ目的で使用されています。実際の環境では、強力で一意のパスワードまたはSSH鍵認証を使用してください。

  1. cloud-initを使用してパスワードを設定する (クラウドイメージ):

    cloudInitNoCloud:
  userData: |
    #cloud-config
    user: fedora
    password: mypassword
    chpasswd:
      expire: false

  2. または、SSH鍵認証を使用し、コンソールで鍵を使用してアクセスする

  3. VM内のcloud-initログを確認してユーザー作成エラーがないか確認する

方法2: VNCアクセス

問題: virtctl vncコマンドが機能しない

症状:

$ virtctl vnc my-vm -n my-namespace
Error: VNC is not configured for this VM

診断:

# グラフィックスが有効か確認する
oc get vm my-vm -n my-namespace -o jsonpath='{.spec.template.spec.domain.devices.autoattachGraphicsDevice}'

# VMデバイス構成を確認する
oc get vm my-vm -n my-namespace -o yaml | grep -A 10 "devices:"

一般的な原因: - `autoattachGraphicsDevice: false`によってグラフィックスデバイスが無効になっている - VM仕様でグラフィックスデバイスを明示的に除外している

解決策: グラフィックスデバイスが有効になっていることを確認します (OpenShift Virtualizationではデフォルトで有効になっています)。明示的に無効になっている場合は、`autoattachGraphicsDevice: false`設定を削除するか、`true`に設定します。

問題: VNCビューアーが接続できない

症状: virtctl vncコマンドが実行されますが、VNCビューアーが接続できません。

診断:

# virtctl出力からポート番号を確認する
virtctl vnc my-vm -n my-namespace
# 出力: "VNC connection on port 5900"

# ポートがリスニングしているか確認する
netstat -tuln | grep 5900

一般的な原因: - ファイアウォールがVNCポートをブロックしている - VNCビューアーがインストールされていないか、正しいビューアーではない - ポート競合

解決策:

  1. VNCビューアーをインストールする:

    # Linux
sudo dnf install tigervnc

# macOS
brew install tiger-vnc

  2. localhost:5900または127.0.0.1:5900に接続する

  3. ポート競合がある場合は、virtctlが次の利用可能なポートを使用します (5901、5902など)
