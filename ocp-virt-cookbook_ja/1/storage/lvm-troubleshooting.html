LVMストレージオペレーターのトラブルシューティング

概要

このガイドでは、OpenShift における LVM Storage オペレーターのトラブルシューティング手順について説明します。これらのコマンドを使用して、LVM ストレージのプロビジョニング、ボリュームグループ、および永続ボリューム要求に関する一般的な問題を診断し、解決してください。

クイックヘルスチェック

LVM ストレージが正しく動作していることを確認するには、このヘルスチェックを実行します。

# 1. Check LVMCluster status
oc get lvmcluster -n openshift-lvm-storage

# 2. Check all pods are running
oc get pods -n openshift-lvm-storage

# 3. Check StorageClass exists
oc get storageclass | grep lvms

# 4. Check volume group on node
oc debug node/<node-name> -- chroot /host vgs

# 5. Check thin pool
oc debug node/<node-name> -- chroot /host lvs

正常な出力の例:

  • LVMCluster: STATUS: Ready

  • Pods: すべて Running で再起動なし

  • StorageClass: lvms-vg1 が存在する

  • VG: 利用可能なスペースがあるボリュームグループを表示

  • LV: データ/メタ使用率の低いシンプールを表示

コンポーネント別の確認コマンド

LVMCluster のステータス

基本ステータスの確認

oc get lvmcluster -n openshift-lvm-storage

正常な出力:

NAME         STATUS
lvmcluster   Ready

異常を示すインジケーター:

  • ステータスが NotReady、Degraded、または Error を示す

  • リソースが見つからない

詳細ステータスの確認

oc get lvmcluster lvmcluster -n openshift-lvm-storage -o yaml

確認すべき点:

status:
  conditions:
  - status: "True"
    type: ResourcesAvailable
    message: Reconciliation is complete and all the resources are available
  - status: "True"
    type: VolumeGroupsReady
    message: All the VGs are ready
  deviceClassStatuses:
  - name: vg1
    nodeStatus:
    - devices:
      - /dev/sda
      status: Ready
  ready: true
  state: Ready

主要なフィールド:

  • status.conditions: ResourcesAvailable: True と VolumeGroupsReady: True を示す必要がある

  • deviceClassStatuses[].nodeStatus[].status: Ready である必要がある

  • deviceClassStatuses[].nodeStatus[].devices: 設定したディスクをリスト表示する

  • ready: true と state: Ready

デバイスの検出と除外の確認

oc get lvmcluster lvmcluster -n openshift-lvm-storage -o yaml | grep -A 50 "deviceClassStatuses"

出力例:

deviceClassStatuses:
- name: vg1
  nodeStatus:
  - devices:
    - /dev/sda
    excluded:
    - name: /dev/sdc
      reasons:
      - /dev/sdc has children block devices and could not be considered
      - /dev/sdc is not part of the device selector
    status: Ready

これは、どのデバイスが選択され、なぜ他のデバイスが除外されたかを示します。

Pod のステータス

すべての Pod の確認

oc get pods -n openshift-lvm-storage

正常な出力:

NAME                             READY   STATUS    RESTARTS   AGE
lvms-operator-864b6975bb-kxt4t   1/1     Running   0          10m
vg-manager-xxxxx                 1/1     Running   0          5m

期待される Pod:

  • lvms-operator: 常に存在し、LVM ストレージのライフサイクルを管理する

  • vg-manager: ノードごとに 1 つ存在し、そのノード上のボリュームグループを管理する

  • topolvm-controller: Single Node OpenShift (SNO) には存在しない場合がある

Pod の詳細の確認

# Get pod details
oc describe pod -n openshift-lvm-storage <pod-name>

# Check specific pod logs
oc logs -n openshift-lvm-storage <pod-name>

# Check vg-manager logs (most useful for device issues)
oc logs -n openshift-lvm-storage daemonset/vg-manager

一般的なログエラー:

  • "device is in use" - ディスクがすでにマウントされているか、システムによって使用されている

  • "cannot wipe device" - forceWipeDevicesAndDestroyAllData: true が必要

  • "device not found" - 誤ったデバイスパスが指定されている

  • "no devices found" - デバイスセレクターが利用可能なデバイスと一致しない

オペレーターインストールの確認

# Check ClusterServiceVersion
oc get csv -n openshift-lvm-storage

# Check subscription
oc get subscription -n openshift-lvm-storage

# Check install plan
oc get installplan -n openshift-lvm-storage

ストレージリソース

StorageClass の確認

# List all StorageClasses
oc get storageclass

# Get specific LVM StorageClass
oc get storageclass lvms-vg1 -o yaml

正常な出力:

NAME       PROVISIONER   RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
lvms-vg1   topolvm.io    Delete          WaitForFirstConsumer   true                   10m

主要なパラメーター:

  • PROVISIONER: topolvm.io - 正しいプロビジョナー

  • VOLUMEBINDINGMODE: WaitForFirstConsumer - 効率的なバインディング

  • ALLOWVOLUMEEXPANSION: true - ボリューム拡張が有効

StorageClass がデフォルトであるかどうかの確認

oc get storageclass lvms-vg1 -o jsonpath='{.metadata.annotations.storageclass\.kubernetes\.io/is-default-class}'

出力:

  • true - これはデフォルトの StorageClass である

  • 出力なし、または false - デフォルトではない

PersistentVolume の確認

# List all PVs
oc get pv

# Check PVs using LVM storage
oc get pv | grep topolvm

ボリュームグループの検証

ボリュームグループステータスの確認

oc debug node/<node-name> -- chroot /host vgs

正常な出力:

VG   #PV #LV #SN Attr   VSize   VFree
vg1    1   1   0 wz--n- 558.37g <55.84g

確認すべき点:

  • VG 名が LVMCluster 設定と一致すること (例: vg1)

  • VSize がディスクの合計サイズを示す

  • VFree が利用可能なスペースを示す

  • Attr に wz—​n- (書き込み可能、サイズ変更可能、通常) が含まれていること

ボリュームグループ詳細の確認

oc debug node/<node-name> -- chroot /host vgdisplay vg1

出力例:

--- Volume group --- VG Name vg1 System ID Format lvm2 Metadata Areas 1 Metadata Sequence No 2 VG Access read/write VG Status resizable MAX LV 0 Cur LV 1 Open LV 0 Max PV 0 Cur PV 1 Act PV 1 VG Size 558.37 GiB PE Size 4.00 MiB Total PE 142942 Alloc PE / Size 128517 / 502.04 GiB Free PE / Size 14425 / 56.35 GiB VG UUID abc123…​

正常性を示す主要なインジケーター:

* `VG Status: resizable`
* `VG Access: read/write`
* `Free PE / Size` が利用可能なスペースを示す

=== シンプールの検証

==== シンプールステータスの確認

[source,bash,role=execute]

oc debug node/<node-name> — chroot /host lvs

正常な出力:

LV VG Attr LSize Pool Origin Data% Meta% thin-pool-1 vg1 twi-a-tz-- <502.04g 0.00 6.76

属性の内訳 (twi-a-tz--) :

* `t` - シンプール
* `w` - 書き込み可能
* `i` - 継承済み
* `a` - アクティブ
* `t` - シン
* `z` - ゼロ

Data% および Meta% のしきい値:

* **Data%**: シンプールのデータ使用量を示す
  ** `< 80%` - 正常
  ** `80-90%` - 厳密に監視する
  ** `> 90%` - 拡張またはクリーンアップを検討する
* **Meta%**: メタデータ使用量を示す
  ** `< 80%` - 正常
  ** `> 80%` - メタデータ拡張が必要になる場合がある

==== シンプール詳細の確認

[source,bash,role=execute]

oc debug node/<node-name> — chroot /host lvdisplay vg1/thin-pool-1

=== 物理ボリュームの検証

==== 物理ボリュームの確認

[source,bash,role=execute]

oc debug node/<node-name> — chroot /host pvs

正常な出力:

PV VG Fmt Attr PSize PFree /dev/sda vg1 lvm2 a-- 558.37g <55.84g

属性の内訳 (a--) :

* `a` - 割り当て可能
* `-` - エクスポートなし
* `-` - 欠落なし

==== 物理ボリューム詳細の確認

[source,bash,role=execute]

oc debug node/<node-name> — chroot /host pvdisplay /dev/sda

==== デバイスパスの確認

[source,bash,role=execute]

Check device exists

oc debug node/<node-name> — chroot /host ls -la /dev/sda

Check device is a block device

oc debug node/<node-name> — chroot /host lsblk /dev/sda

Check device is not mounted

oc debug node/<node-name> — chroot /host mount | grep /dev/sda

NOTE: デバイスがマウントされている場合、LVM ストレージには使用できません。アンマウントするか、別のデバイスを選択してください。

=== ディスクとデバイスの情報

==== すべてのブロックデバイスを一覧表示

[source,bash,role=execute]

oc debug node/<node-name> — chroot /host lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT,MODEL

出力例:

NAME SIZE TYPE FSTYPE MOUNTPOINT MODEL sda 558.4G disk LVM2_member PERC H730 Adp sdc 558.4G disk PERC H730 Adp |-sdc1 1M part |-sdc2 127M part vfat |-sdc3 384M part ext4 /boot `-sdc4 557.9G part xfs /sysroot

使用可能なディスクを識別する方法:

* MOUNTPOINT なし - ディスクは使用中ではない
* FSTYPE なし、または `LVM2_member` を示す (ワイプが意図されている場合)
* MOUNTPOINT あり - ディスクはシステムによって使用中
* パーティション (子) がある - ワイプが必要な場合がある

==== ディスクシリアル番号の確認

[source,bash,role=execute]

oc debug node/<node-name> — chroot /host lsblk -d -o NAME,SIZE,SERIAL

これを使用して、正しいディスクを対象としていることを確認します。

==== デバイスの永続パスの確認

[source,bash,role=execute]

By path

oc debug node/<node-name> — chroot /host ls -la /dev/disk/by-path/ | grep sda

By ID

oc debug node/<node-name> — chroot /host ls -la /dev/disk/by-id/ | grep sda

== 一般的な問題と解決策

=== 問題 1: LVMCluster が "NotReady" を示す

症状:

NAME STATUS lvmcluster NotReady

診断:

[source,bash,role=execute]

Check detailed status

oc get lvmcluster lvmcluster -n openshift-lvm-storage -o yaml | grep -A 20 "conditions"

Check vg-manager logs

oc logs -n openshift-lvm-storage daemonset/vg-manager

一般的な原因と解決策:

==== デバイスが見つからない、またはパスが間違っている

[source,bash,role=execute]

Verify device exists

oc debug node/<node-name> — chroot /host ls -la /dev/sda

解決策: LVMCluster を正しいデバイスパスで更新する

==== デバイスに既存のデータ/ファイルシステムがある

[source,bash,role=execute]

Check for existing filesystem

oc debug node/<node-name> — chroot /host lsblk -f /dev/sda

解決策: `forceWipeDevicesAndDestroyAllData: true` を deviceSelector に追加する

==== デバイスが使用中である

[source,bash,role=execute]

Check if mounted

oc debug node/<node-name> — chroot /host mount | grep /dev/sda

解決策: デバイスをアンマウントするか、別のディスクを使用する

=== 問題 2: ボリュームグループが作成されない

症状:

* LVMCluster は存在するが、`vgs` がボリュームグループを表示しない
* deviceClassStatuses にデバイスが表示されない

診断:

[source,bash,role=execute]

Check if disk was wiped

oc debug node/<node-name> — chroot /host pvs

Check vg-manager logs for errors

oc logs -n openshift-lvm-storage daemonset/vg-manager --tail=100

Check device visibility

oc debug node/<node-name> — chroot /host lsblk -o NAME,SIZE,TYPE,FSTYPE

解決策:

==== デバイスが選択されない

* deviceSelector パスが実際のデバイスと一致することを確認する
* デバイスが除外されていないことを確認する (deviceClassStatuses.excluded を参照)

==== パーミッションの問題

* vg-manager Pod が必要な特権を持っていることを確認する
* ノードログで SELinux の拒否を確認する

==== デバイスのワイプに失敗した

* `forceWipeDevicesAndDestroyAllData: true` を設定する
* デバイスを手動でワイプし、オペレーターを再起動する

=== 問題 3: PVC が Pending 状態のままになる

症状:

NAME STATUS VOLUME CAPACITY ACCESS MODES STORAGECLASS AGE test-pvc Pending lvms-vg1 5m

診断:

[source,bash,role=execute]

Check PVC events

oc describe pvc <pvc-name>

Check if pod using PVC exists

oc get pods --all-namespaces -o wide | grep <pvc-name>

Check available space in VG

oc debug node/<node-name> — chroot /host vgs

一般的な原因:

==== WaitForFirstConsumer - Pod がまだスケジュールされていない

* **通常の動作**: PVC は、それを使用する Pod がスケジュールされるまで Pending のままになる
* **解決策**: PVC を使用する Pod を作成する

==== スペース不足

[source,bash,role=execute]

Check VG free space

oc debug node/<node-name> — chroot /host vgdisplay vg1 | grep "Free"

解決策: 未使用の PV を削除するか、ボリュームグループを拡張する

==== シンプールがいっぱい

[source,bash,role=execute]

Check thin pool usage

oc debug node/<node-name> — chroot /host lvs | grep thin-pool

解決策: ボリュームをクリーンアップするか、シンプールのサイズを増やす

==== ノードセレクターの不一致

[source,bash,role=execute]

Check pod node selector vs LVMCluster node selector

oc get pod <pod-name> -o yaml | grep nodeSelector oc get lvmcluster lvmcluster -n openshift-lvm-storage -o yaml | grep -A 5 nodeSelector

=== 問題 4: シンプールの Data% または Meta% が高い

症状:

LV VG Attr LSize Data% Meta% thin-pool-1 vg1 twi-a-tz-- 502.04g 92.00 85.00

診断:

[source,bash,role=execute]

Check thin pool details

oc debug node/<node-name> — chroot /host lvs -a | grep thin-pool

List all thin volumes

oc debug node/<node-name> — chroot /host lvs -a -o lv_name,lv_size,data_percent,pool_lv | grep thin-pool

Check which PVs are consuming space

oc get pv -o custom-columns=NAME:.metadata.name,SIZE:.spec.capacity.storage,STORAGECLASS:.spec.storageClassName | grep lvms

解決策:

==== 未使用の PVC を削除する

[source,bash,role=execute]

Find PVCs not in use

oc get pvc --all-namespaces

Delete PVC

oc delete pvc <pvc-name> -n <namespace>

==== シンプールを拡張する (VG に空き容量がある場合)

[source,bash,role=execute]

Check VG free space

oc debug node/<node-name> — chroot /host vgs

Extend thin pool (if needed manually)

Usually handled automatically by LVM operator

==== プロビジョニング比率の調整

LVMCluster を編集して `overprovisionRatio` を減らす

=== 問題 5: デバイスのワイプに失敗した

症状:

* LVMCluster は作成されたが、デバイスが使用されていない
* ログに "cannot wipe device" または "device busy" が表示される

診断:

[source,bash,role=execute]

Check current filesystem/partition

oc debug node/<node-name> — chroot /host lsblk -f /dev/sda

Check for LVM signatures

oc debug node/<node-name> — chroot /host pvs | grep /dev/sda

Check if device is open

oc debug node/<node-name> — chroot /host lsof | grep /dev/sda

解決策:

[source,bash,role=execute]

Ensure forceWipeDevicesAndDestroyAllData is set

oc edit lvmcluster lvmcluster -n openshift-lvm-storage

Add under deviceSelector:

forceWipeDevicesAndDestroyAllData: true

=== 問題 6: vg-manager Pod が CrashLooping になる

症状:

NAME READY STATUS RESTARTS AGE vg-manager-xxx 0/1 CrashLoopBackOff 5 3m

診断:

[source,bash,role=execute]

Check pod logs

oc logs -n openshift-lvm-storage <vg-manager-pod> --previous

Check pod events

oc describe pod -n openshift-lvm-storage <vg-manager-pod>

Check node resources

oc describe node <node-name>

一般的な原因:

==== デバイスのパーミッション

* vg-manager にデバイスへのアクセス権がない可能性がある
* ノードログで SELinux の拒否を確認する

==== デバイスの欠落

* 設定のデバイスパスが存在しない
* デバイスが削除されたか、名前が変更された

==== リソースの制約

* ノードのメモリーまたは CPU が不足している
* ノードの容量を確認する

=== 問題 7: StorageClass がデフォルトではない

症状:

* storageClassName が明示的に指定されていない PVC はバインドに失敗する
* 警告: "no default deviceClass was specified"

診断:

[source,bash,role=execute]

Check if any StorageClass is default

oc get storageclass | grep default

Check lvms StorageClass annotations

oc get storageclass lvms-vg1 -o yaml | grep is-default-class

解決策:

==== LVM StorageClass をデフォルトに設定する

[source,bash,role=execute]

oc patch storageclass lvms-vg1 -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'

==== または LVMCluster でデフォルトを設定する

[source,bash,role=execute]

oc edit lvmcluster lvmcluster -n openshift-lvm-storage

Add under deviceClasses:

  • name: vg1 default: true

== 診断スクリプト

これをシェルスクリプトとして保存し、LVM ストレージの問題をすばやく診断してください。

[source,bash,role=execute]

#!/bin/bash

NAMESPACE="openshift-lvm-storage" NODE_NAME="<your-node-name>"

echo "=== LVM ストレージ診断レポート ===" echo ""

echo "1. LVMCluster ステータス:" oc get lvmcluster -n $NAMESPACE echo ""

echo "2. Pods ステータス:" oc get pods -n $NAMESPACE echo ""

echo "3. StorageClass:" oc get storageclass | grep lvms echo ""

echo "4. PVs:" oc get pv | grep topolvm echo ""

echo "5. ボリュームグループ:" oc debug node/$NODE_NAME — chroot /host vgs echo ""

echo "6. 論理ボリューム:" oc debug node/$NODE_NAME — chroot /host lvs echo ""

echo "7. 物理ボリューム:" oc debug node/$NODE_NAME — chroot /host pvs echo ""

echo "8. ブロックデバイス:" oc debug node/$NODE_NAME — chroot /host lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT echo ""

echo "9. 最近の vg-manager ログ:" oc logs -n $NAMESPACE daemonset/vg-manager --tail=20 echo ""

echo "=== 診断レポートの終了 ==="

== まとめ

このガイドでは、以下の内容を学びました。

* LVMCluster、Pod、StorageClass、ボリュームグループ、シンプールを確認するためのクイックヘルスチェックを実行する方法
* LVMCluster の NotReady、ボリュームグループの欠落、Pending 状態の PVC など、一般的な問題を診断する方法
* `vgs`、`lvs`、`pvs` コマンドを使用してボリュームグループ、シンプール、および物理ボリュームのステータスを確認する方法
* デバイスのワイプ失敗、vg-manager の CrashLoopBackOff、および StorageClass のデフォルト設定をトラブルシューティングする方法
* 診断スクリプトを使用して、LVM ストレージのヘルス情報を迅速に収集する方法
